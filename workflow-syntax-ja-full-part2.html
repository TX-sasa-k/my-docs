ワークフローの YAML 構文（日本語版） — Part 2

対象範囲：permissions 〜 ジョブレベルの構文（jobs.<job_id>.needs まで）

本ページは GitHub 公式ドキュメント「ワークフローの構文」の主要キーワードを、分かりやすい日本語でまとめ直した解説です。正確な仕様は必ず公式ドキュメントをご確認ください。

permissions

permissions を使用して、GITHUB_TOKEN に付与する権限を変更できます。ワークフロー全体に対して設定することも、個々のジョブに対して設定することもできます。最小権限の原則に従い、必要なアクセス権だけを付与することが推奨されます。

設定できる値は read（読み取り）、write（書き込み）、none（権限なし）です。あるスコープに write を指定すると、read も暗黙的に含まれます。

permissions:
  actions: read
  contents: write
  issues: write
  pull-requests: write

すべてのスコープにまとめて権限を付与・剥奪することもできます。

# すべてのスコープを読み取り専用にする
permissions: read-all

# すべての権限を剥奪する
permissions: {}
メモ：利用可能なスコープには actions、checks、contents、deployments、id-token、issues、packages、pages、pull-requests、security-events などがあります。

env

env を使用して、ワークフロー内で利用できる環境変数を定義します。設定する場所によってスコープ（適用範囲）が異なります。

  • ワークフローのトップレベル → すべてのジョブのすべてのステップで利用可能
  • jobs.<job_id>.env → そのジョブ内のすべてのステップで利用可能
  • jobs.<job_id>.steps[*].env → その単一ステップでのみ利用可能

同じ名前の変数が複数の場所で定義されている場合、より狭いスコープ（ステップ > ジョブ > ワークフロー）の値が優先されます。

env:
  SERVER: production

jobs:
  build:
    env:
      LOG_LEVEL: info
    runs-on: ubuntu-latest
    steps:
      - name: 変数を表示
        env:
          FIRST_NAME: Mona
        run: echo "$SERVER / $LOG_LEVEL / $FIRST_NAME"

defaults

defaults を使用して、ワークフロー内のすべてのジョブに適用されるデフォルト設定を定義します。ジョブ単位で jobs.<job_id>.defaults を設定することもできます。

defaults.run

defaults.run を使用して、ワークフロー内のすべての run ステップに対するデフォルトの shell と working-directory（作業ディレクトリ）を設定できます。

defaults:
  run:
    shell: bash
    working-directory: ./scripts
メモ：このデフォルトが影響するのは run を使うステップのみで、uses でアクションを呼び出すステップには影響しません。

concurrency

concurrency を使用して、同じ並行実行グループを使うワークフロー実行やジョブを一度に1つだけに制限できます。group には任意の文字列や式を指定でき、同じグループ名の実行が同時に走らないように調整されます。

cancel-in-progress: true を指定すると、同じグループで進行中の実行をキャンセルして新しい実行を開始します。

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

たとえば上記の例では、同じブランチで新しいワークフローが開始されると、進行中の古い実行が自動的にキャンセルされます。デプロイなど「最新の実行だけを残したい」場面で便利です。

jobs

ワークフローの実行は、1 つ以上の jobs（ジョブ）で構成されます。ジョブはデフォルトで並行（同時）に実行されます。順番に実行したい場合は jobs.<job_id>.needs を使って依存関係を定義します。

各ジョブは、runs-on で指定したランナー環境で実行されます。

jobs:
  my_first_job:
    name: 1つ目のジョブ
    runs-on: ubuntu-latest
  my_second_job:
    name: 2つ目のジョブ
    runs-on: ubuntu-latest

jobs.<job_id>

各ジョブには一意の ID（job_id）を付けます。job_id は文字列で、英字または _ で始め、英数字・-・_ のみを使用できます。

jobs.<job_id>.name

GitHub の UI に表示されるジョブ名です。省略した場合は job_id がそのまま表示名になります。

jobs.<job_id>.runs-on

runs-on でジョブを実行するランナーの種類を指定します。これは必須項目です。GitHub ホストランナー、大規模ランナー、セルフホストランナーを指定できます。

# GitHub ホストランナー（単一ラベル）
runs-on: ubuntu-latest

複数のラベルを指定すると、すべてのラベルに一致するランナーで実行されます（主にセルフホストランナーのグループ指定に使用）。

runs-on: [self-hosted, linux, x64]
主なラベル例説明
ubuntu-latest最新の Ubuntu 環境
windows-latest最新の Windows 環境
macos-latest最新の macOS 環境
self-hostedセルフホストランナー
注意：-latest ランナーは、GitHub が提供する最新の安定版イメージを指します。再現性が重要な場合は、ubuntu-22.04 のようにバージョンを固定することを検討してください。

jobs.<job_id>.permissions

ワークフロー全体ではなく、特定のジョブに対して GITHUB_TOKEN の権限を設定します。書式はワークフローレベルの permissions と同じです。

jobs.<job_id>.needs

needs を使用して、そのジョブを開始する前に正常完了している必要がある他のジョブを指定します。これによりジョブの実行順序（依存関係）を定義できます。

jobs:
  job1:
    runs-on: ubuntu-latest
  job2:
    needs: job1          # job1 の完了後に実行
    runs-on: ubuntu-latest
  job3:
    needs: [job1, job2]  # job1 と job2 の両方の完了後に実行
    runs-on: ubuntu-latest

この例では、job1 → job2 → job3 の順で実行されます。依存先のジョブが失敗すると、後��のジョブは原則スキップされます。失敗しても続行したい場合は、後続ジョブで if: always() などの条件を併用します。

jobs.<job_id>.if

if 条件を使用して、条件が満たされた場合にのみジョブを実行できます。式を書く際、${{ }} は省略可能です（GitHub が自動的に式として評価します）。

jobs:
  deploy:
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
