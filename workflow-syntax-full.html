GitHub Actions ワークフロー構文（日本語）

日本語訳: ワークフロー向け YAML 構文について

日本語訳: ワークフローファイルは YAML 構文を使用し、拡張子は .yml または .yaml のいずれかである必要があります。 YAML が初めてで詳しく学びたい場合は、Learn YAML in Y minutes を参照してください。

日本語訳: ワークフローファイルはリポジトリの .github/ワークフローs ディレクトリに保存する必要があります。

name

日本語訳: ワークフローの名前です。 GitHub displays the names of your ワークフローs under your repository's "Actions" tab. もし omit name, GitHub displays the ワークフロー file path relative to the root of the repository.

run-name

日本語訳: The name for ワークフロー runs generated from the ワークフロー. GitHub displays the ワークフロー run name in the list of ワークフロー runs on your repository's "Actions" tab. If run-name is omitted or is only whitespace, then the run name is set to イベント-specific information for the ワークフロー run. 例えば、 for a ワークフロー triggered by a push or pull_request イベント, it is set as the commit message or the title of the pull request.

日本語訳: この値には式を含めることができ、github および inputs コンテキストを参照できます。

日本語訳: run-name の例

run-name: Deploy to ${{ inputs.deploy_target }} by @${{ github.actor }}
on

日本語訳: 〜するには automatically trigger a ワークフロー, use on to define which イベントs can cause the ワークフロー to run. For a list of available イベントs, see Events that trigger ワークフローs.

日本語訳: 次のことができます define single or multiple イベントs that can trigger a ワークフロー, or set a time schedule. 次のことができます also restrict the execution of a ワークフロー to only occur for specific files, タグs, or ブランチ changes. These options are described in the following sections.

日本語訳: 単一イベントの使用

日本語訳: 例えば、 a ワークフロー with the following on value will run when a push is made to any ブランチ in the ワークフロー's repository:

on: push

日本語訳: 複数イベントの使用

日本語訳: 次のことができます specify a single イベント or multiple イベントs. 例えば、 a ワークフロー with the following on value will run when a push is made to any ブランチ in the repository or when someone forks the repository:

on: [push, fork]

日本語訳: もし specify multiple イベントs, only one of those イベントs needs to occur to trigger your ワークフロー. If multiple triggering イベントs for your ワークフロー occur at the same time, multiple ワークフロー runs will be triggered.

日本語訳: アクティビティタイプの使用

日本語訳: Some イベントs have activity types that give you more control over when your ワークフロー should run. Use on.<イベント_name>.types to define the type of イベント activity that will trigger a ワークフロー run.

日本語訳: 例えば、 the issue_comment イベント has the created, edited, and deleted activity types. もし r ワークフロー triggers on the label イベント, it will run whenever a label is created, edited, or deleted. もし specify the created activity type for the label イベント, your ワークフロー will run when a label is created but not when a label is edited or deleted.

on:
  label:
    types:
      - created

日本語訳: もし specify multiple activity types, only one of those イベント activity types needs to occur to trigger your ワークフロー. If multiple triggering イベント activity types for your ワークフロー occur at the same time, multiple ワークフロー runs will be triggered. 例えば、 the following ワークフロー triggers when an issue is opened or labeled. If an issue with two labels is opened, three ワークフロー runs will start: one for the issue opened イベント and two for the two issue labeled イベントs.

on:
  issues:
    types:
      - opened
      - labeled

日本語訳: 詳細については about each イベント and their activity types, see Events that trigger ワークフローs.

日本語訳: フィルターの使用

日本語訳: Some イベントs have フィルターs that give you more control over when your ワークフロー should run.

日本語訳: 例えば、 the push イベント has a ブランチes フィルター that causes your ワークフロー to run only when a push to a ブランチ that matches the ブランチes フィルター occurs, instead of when any push occurs.

on:
  push:
    branches:
      - main
      - 'releases/**'

日本語訳: 複数イベントでのアクティビティタイプとフィルターの併用

日本語訳: もし specify activity types or フィルターs for an イベント and your ワークフロー triggers on multiple イベントs, you must configure each イベント separately. You must append a colon (:) to all イベントs, including イベントs without configuration.

日本語訳: 例えば、 a ワークフロー with the following on value will run when:

日本語訳: A label is created

日本語訳: A push is made to the main ブランチ in the repository

日本語訳: A push is made to a GitHub Pages-enabled ブランチ

on:
  label:
    types:
      - created
  push:
    branches:
      - main
  page_build:

日本語訳: on.<イベント_name>.types

日本語訳: Use on.<イベント_name>.types to define the type of activity that will trigger a ワークフロー run. Most GitHub イベントs are triggered by more than one type of activity. 例えば、 the label is triggered when a label is created, edited, or deleted. The types keyword enables you to narrow down activity that causes the ワークフロー to run. 〜する場合、 only one activity type triggers a webhook イベント, the types keyword is unnecessary.

日本語訳: 次のことができます use an array of イベント types. 詳細については about each イベント and their activity types, see Events that trigger ワークフローs.

on:
  label:
    types: [created, edited]

日本語訳: on.<pull_request|pull_request_target>.<ブランチes|ブランチes-ignore>

日本語訳: 〜する場合、 using the pull_request and pull_request_target イベントs, you can configure a ワークフロー to run only for pull requests that target specific ブランチes.

日本語訳: Use the ブランチes フィルター when you want to include ブランチ name patterns or when you want to both include and exclude ブランチ names patterns. Use the ブランチes-ignore フィルター when you only want to exclude ブランチ name patterns. 次のことができますnot use both the ブランチes and ブランチes-ignore フィルターs for the same イベント in a ワークフロー.

日本語訳: もし define both ブランチes/ブランチes-ignore and paths/paths-ignore, the ワークフロー will only run when both フィルターs are satisfied.

日本語訳: The ブランチes and ブランチes-ignore keywords accept glob patterns that use characters like *, **, +, ?, ! and others to match more than one ブランチ name. If a name contains any of these characters and you want a literal match, you need to escape each of these special characters with \. 詳細については about glob patterns, see the Workflow syntax for GitHub Actions.

Example: Including branches

日本語訳: The patterns defined in ブランチes are evaluated against the Git ref's name. 例えば、 the following ワークフロー would run whenever there is a pull_request イベント for a pull request targeting:

日本語訳: A ブランチ named main (refs/heads/main)

日本語訳: A ブランチ named mona/octocat (refs/heads/mona/octocat)

日本語訳: A ブランチ whose name starts with releases/, like releases/10 (refs/heads/releases/10)

on:
  pull_request:
    # Sequence of patterns matched against refs/heads
    branches:
      - main
      - 'mona/octocat'
      - 'releases/**'

日本語訳: If a ワークフロー is skipped due to ブランチ フィルターing, path フィルターing, or a commit message, then checks associated with that ワークフロー will remain in a "Pending" state. A pull request that requires those checks to be successful will be blocked from merging.

Example: Excluding branches

日本語訳: 〜する場合、 a pattern matches the ブランチes-ignore pattern, the ワークフロー will not run. The patterns defined in ブランチes-ignore are evaluated against the Git ref's name. 例えば、 the following ワークフロー would run whenever there is a pull_request イベント unless the pull request is targeting:

日本語訳: A ブランチ named mona/octocat (refs/heads/mona/octocat)

日本語訳: A ブランチ whose name matches releases/**-alpha, like releases/beta/3-alpha (refs/heads/releases/beta/3-alpha)

on:
  pull_request:
    # Sequence of patterns matched against refs/heads
    branches-ignore:
      - 'mona/octocat'
      - 'releases/**-alpha'
Example: Including and excluding branches

日本語訳: 次のことができますnot use ブランチes and ブランチes-ignore to フィルター the same イベント in a single ワークフロー. もし want to both include and exclude ブランチ patterns for a single イベント, use the ブランチes フィルター along with the ! character to indicate which ブランチes should be excluded.

日本語訳: もし define a ブランチ with the ! character, you must also define at least one ブランチ without the ! character. もし only want to exclude ブランチes, use ブランチes-ignore instead.

日本語訳: The order that you define patterns matters.

日本語訳: A matching negative pattern (prefixed with !) after a positive match will exclude the Git ref.

日本語訳: A matching positive pattern after a negative match will include the Git ref again.

日本語訳: The following ワークフロー will run on pull_request イベントs for pull requests that target releases/10 or releases/beta/mona, but not for pull requests that target releases/10-alpha or releases/beta/3-alpha because the negative pattern !releases/**-alpha follows the positive pattern.

on:
  pull_request:
    branches:
      - 'releases/**'
      - '!releases/**-alpha'

日本語訳: Use the ブランチes フィルター when you want to include ブランチ name patterns or when you want to both include and exclude ブランチ names patterns. Use the ブランチes-ignore フィルター when you only want to exclude ブランチ name patterns. 次のことができますnot use both the ブランチes and ブランチes-ignore フィルターs for the same イベント in a ワークフロー.

日本語訳: Use the タグs フィルター when you want to include タグ name patterns or when you want to both include and exclude タグ names patterns. Use the タグs-ignore フィルター when you only want to exclude タグ name patterns. 次のことができますnot use both the タグs and タグs-ignore フィルターs for the same イベント in a ワークフロー.

日本語訳: on.<push|pull_request|pull_request_target>.<paths|paths-ignore>

日本語訳: 〜する場合、 using the push and pull_request イベントs, you can configure a ワークフロー to run based on what file paths are changed. Path フィルターs are not evaluated for pushes of タグs.

日本語訳: Use the paths フィルター when you want to include file path patterns or when you want to both include and exclude file path patterns. Use the paths-ignore フィルター when you only want to exclude file path patterns. 次のことができますnot use both the paths and paths-ignore フィルターs for the same イベント in a ワークフロー. もし want to both include and exclude path patterns for a single イベント, use the paths フィルター prefixed with the ! character to indicate which paths should be excluded.

日本語訳: メモ

日本語訳: The order that you define paths patterns matters:

日本語訳: A matching negative pattern (prefixed with !) after a positive match will exclude the path.

日本語訳: A matching positive pattern after a negative match will include the path again.

日本語訳: もし define both ブランチes/ブランチes-ignore and paths/paths-ignore, the ワークフロー will only run when both フィルターs are satisfied.

日本語訳: The paths and paths-ignore keywords accept glob patterns that use the * and ** wildcard characters to match more than one path name. 詳細については, see the Workflow syntax for GitHub Actions.

Example: Including paths

日本語訳: If at least one path matches a pattern in the paths フィルター, the ワークフロー runs. 例えば、 the following ワークフロー would run anytime you push a JavaScript file (.js).

on:

日本語訳: Git diff comparisons

日本語訳: メモ

日本語訳: もし push more than 1,000 commits, or if GitHub does not generate the diff due to a timeout, the ワークフロー will always run.

日本語訳: The フィルター determines if a ワークフロー should run by evaluating the changed files and running them against the paths-ignore or paths list. If there are no files changed, the ワークフロー will not run.

日本語訳: GitHub generates the list of changed files using two-dot diffs for pushes and three-dot diffs for pull requests:

日本語訳: Pull requests: Three-dot diffs are a comparison between the most recent version of the topic ブランチ and the commit where the topic ブランチ was last synced with the base ブランチ.

日本語訳: Pushes to existing ブランチes: A two-dot diff compares the head and base SHAs directly with each other.

日本語訳: Pushes to new ブランチes: A two-dot diff against the parent of the ancestor of the deepest commit pushed.

日本語訳: メモ

日本語訳: Diffs are limited to 300 files. If there are files changed that aren't matched in the first 300 files returned by the フィルター, the ワークフロー will not run. You may need to create more specific フィルターs so that the ワークフロー will run automatically.

日本語訳: 詳細については, see About comparing ブランチes in pull requests.

on.schedule

日本語訳: 次のことができます use on.schedule to define a time schedule for your ワークフローs.

日本語訳: Use POSIX cron syntax to schedule ワークフローs to run at specific times. By デフォルト, scheduled ワークフローs run in UTC. 次のことができます optionally specify a timezone using an IANA timezone string for timezone-aware scheduling. Scheduled ワークフローs run on the latest commit on the デフォルト ブランチ. The shortest interval you can run scheduled ワークフローs is once every 5 minutes.

日本語訳: メモ

日本語訳: For schedules that set timezone to a time zone that observes daylight saving time (DST), during DST spring-forward transitions, scheduled ワークフローs in skipped hours advance to the next valid time. 例えば、 a 2:30 AM schedule advances to 3:00 AM.

日本語訳: Cron syntax has five fields separated by a space, and each field represents a unit of time.

日本語訳: ┌───────────── minute (0 - 59)

日本語訳: │ ┌───────────── hour (0 - 23)

日本語訳: │ │ ┌───────────── day of the month (1 - 31)

日本語訳: │ │ │ ┌───────────── month (1 - 12 or JAN-DEC)

日本語訳: │ │ │ │ ┌───────────── day of the week (0 - 6 or SUN-SAT)

日本語訳: │ │ │ │ │

日本語訳: * * * * *

日本語訳: 次のことができます use these operators in any of the five fields:

日本語訳: Operator Description Example

日本語訳: * Any value 15 * * * * runs at every minute 15 of every hour of every day.

日本語訳: , Value list separator 2,10 4,5 * * * runs at minute 2 and 10 of the 4th and 5th hour of every day.

日本語訳: - Range of values 30 4-6 * * * runs at minute 30 of the 4th, 5th, and 6th hour.

日本語訳: / Step values 20/15 * * * * runs every 15 minutes starting from minute 20 through 59 (minutes 20, 35, and 50).

日本語訳: This example triggers the ワークフロー to run at 5:30 AM in the America/New_York timezone every Monday through Friday:

on:
  schedule:
    - cron: '30 5 * * 1-5'
      timezone: "America/New_York"

日本語訳: A single ワークフロー can be triggered by multiple schedule イベントs. Access the schedule イベント that triggered the ワークフロー through the github.イベント.schedule コンテキスト. This example triggers the ワークフロー to run at 5:30 UTC every Monday-Thursday, and 17:30 UTC on Tuesdays and Thursdays, but skips the Not on Monday or Wednesday ステップ on Monday and Wednesday.

on:
  schedule:
    - cron: '30 5 * * 1,3'
    - cron: '30 5,17 * * 2,4'

jobs:
  test_schedule:
    runs-on: ubuntu-latest
    steps:
      - name: Not on Monday or Wednesday
        if: github.event.schedule != '30 5 * * 1,3'
        run: echo "This step will be skipped on Monday and Wednesday"
      - name: Every time
        run: echo "This step will always run"

日本語訳: 詳細については about schedule イベントs, see Events that trigger ワークフローs.

on.workflow_call

日本語訳: Use on.ワークフロー_call to define the inputs and outputs for a reusable ワークフロー. 次のことができます also map the secrets that are available to the called ワークフロー. 詳細については on reusable ワークフローs, see Reuse ワークフローs.

on.workflow_call.inputs

日本語訳: 〜する場合、 using the ワークフロー_call keyword, you can optionally specify inputs that are passed to the called ワークフロー from the caller ワークフロー. 詳細については about the ワークフロー_call keyword, see Events that trigger ワークフローs.

日本語訳: In addition to the standard input parameters that are available, on.ワークフロー_call.inputs requires a type parameter. 詳細については, see on.ワークフロー_call.inputs.<input_id>.type.

日本語訳: If a デフォルト parameter is not set, the デフォルト value of the input is false for a boolean, 0 for a number, and "" for a string.

日本語訳: Within the called ワークフロー, you can use the inputs コンテキスト to refer to an input. 詳細については, see Contexts reference.

日本語訳: If a caller ワークフロー passes an input that is not specified in the called ワークフロー, this results in an error.

日本語訳: Example of on.ワークフロー_call.inputs

    inputs:
      username:
        description: 'A username passed from the caller workflow'
        default: 'john-doe'
        required: false
        type: string

jobs:
  print-username:
    runs-on: ubuntu-latest

    steps:

日本語訳: The 同時実行制御 key is used to group ワークフローs or ジョブs together into a 同時実行制御 group. 〜する場合、 you define a 同時実行制御 key, GitHub Actions ensures that only one ワークフロー or ジョブ with that key runs at any given time. The 同時実行制御 key can be a hard-coded string, or it can be a dynamic 式 that includes コンテキスト variables.

日本語訳: It is possible to define 同時実行制御 conditions in your ワークフロー so that the ワークフロー or ジョブ is part of a 同時実行制御 group.

日本語訳: This means that when a ワークフロー run or ジョブ starts, GitHub will cancel any ワークフロー runs or ジョブs that are already in progress in the same 同時実行制御 group. This is useful in scenarios where you want to prイベント parallel runs for a certain set of a ワークフローs or ジョブs, such as the ones used for deployments to a sタグing environment, in order to prイベント actions that could cause conflicts or consume more resources than necessary.

日本語訳: In this example, ジョブ-1 is part of a 同時実行制御 group named sタグing_environment. This means that if a new run of ジョブ-1 is triggered, any runs of the same ジョブ in the sタグing_environment 同時実行制御 group that are already in progress will be cancelled.

  job-1:
    runs-on: ubuntu-latest
    concurrency:
      group: staging_environment
      cancel-in-progress: true
Alternatively, using a dynamic expression such as concurrency: ci-${{ github.ref }} in your workflow means that the workflow or job would be part of a concurrency group named ci- followed by the reference of the branch or tag that triggered the workflow. In this example, if a new commit is pushed to the main branch while a previous run is still in progress, the previous run will be cancelled and the new one will start:

  group: ci-${{ github.ref }}

日本語訳: By デフォルト, only one ジョブ or ワークフロー run can be pending in a 同時実行制御 group at a time. 〜するには allow multiple runs to queue instead of being canceled, set queue: max.
