ワークフローの YAML 構文（日本語版） — Part 1

対象範囲：概要 〜 on.schedule

ワークフローの YAML 構文について

ワークフローファイルは YAML 構文を使用し、ファイル拡張子は .yml または .yaml でなければなりません。YAML を初めて使う場合は「Learn YAML in Y minutes」を参照してください。

ワークフローファイルはリポジトリの .github/workflows ディレクトリに保存する必要があります。

name

ワークフローの名前です。GitHub はリポジトリの「Actions」タブにワークフロー名を表示します。name を省略した場合、GitHub はリポジトリルートからの相対パスでワークフローファイルのパスを表示します。

run-name

ワークフローから生成されるワークフロー実行の名前です。GitHub はリポジトリの「Actions」タブにあるワークフロー実行一覧にこの名前を表示します。run-name が省略されたり空白のみの場合は、そのワークフロー実行のイベント固有の情報が名前として設定されます。たとえば push または pull_request イベントでトリガーされたワークフローの場合、コミットメッセージまたはプルリクエストのタイトルが設定されます。

この値には式を含めることができ、github および inputs コンテキストを参照できます。

run-name の例

run-name: Deploy to ${{ inputs.deploy_target }} by @${{ github.actor }}

on

ワークフローを自動的にトリガーするには、on を使用して、ワークフローを実行できるイベントを定義します。利用可能なイベントの一覧については、「ワークフローをトリガーするイベント」を参照してください。

ワークフローをトリガーする単一または複数のイベントを定義したり、スケジュールを設定したりできます。また、特定のファイル、タグ、またはブランチへの変更のみでワークフローの実行が発生するよう制限することもできます。

単一イベントの使用

たとえば、次の on 値を持つワークフローは、ワークフローのリポジトリ内の任意のブランチへのプッシュ時に実行されます。

on: push

複数イベントの使用

単一または複数のイベントを指定できます。たとえば、次の on 値を持つワークフローは、リポジトリ内の任意のブランチへのプッシュ時、または誰かがリポジトリをフォークした時に実行されます。

on: [push, fork]

複数のイベントを指定した場合、そのうちの1つが発生すればワークフローがトリガーされます。同時に複数のトリガーイベントが発生した場合は、複数のワークフロー実行が開始されます。

アクティビティタイプの使用

一部のイベントには、ワークフローをいつ実行するかをより細かく制御するためのアクティビティタイプがあります。on.<event_name>.types を使用して、ワークフロー実行をトリガーするイベントアクティビティのタイプを定義します。

たとえば、issue_comment イベントには created、edited、deleted のアクティビティタイプがあります。label イベントでワークフローをトリガーする場合、ラベルが作成・編集・削除されるたびに実行されます。label イベントに created アクティビティタイプを指定すると、ラベルが作成された時のみ実行され、編集や削除では実行されません。

on:
  label:
    types:
      - created

複数のアクティビティタイプを指定した場合は、そのうちの1つが発生すればワークフローがトリガーされます。複数のトリガーイベントアクティビティタイプが同時に発生した場合は、複数のワークフロー実行が開始されます。たとえば、以下のワークフローは issue がオープンまたはラベル付けされた時にトリガーされます。2つのラベルを持つ issue がオープンされた場合、3つのワークフロー実行が開始されます（issue opened イベントに1つ、2つの issue labeled イベントにそれぞれ1つずつ）。

on:
  issues:
    types:
      - opened
      - labeled

フィルターの使用

一部のイベントには、ワークフローをいつ実行するかをより細かく制御するためのフィルターがあります。たとえば、push イベントには branches フィルターがあり、任意のプッシュ時ではなく、branches フィルターに一致するブランチへのプッシュ時のみワークフローが実行されます。

on:
  push:
    branches:
      - main
      - 'releases/**'

複数イベントでのアクティビティタイプとフィルターの使用

イベントにアクティビティタイプやフィルターを指定し、かつワークフローが複数のイベントでトリガーされる場合は、各イベントを個別に設定する必要があります。設定のないイベントも含め、すべてのイベントにコロン（:）を付ける必要があります。

たとえば、以下の on 値を持つワークフローは次の場合に実行されます。

  • ラベルが作成された時
  • リポジトリの main ブランチへのプッシュ時
  • GitHub Pages が有効なブランチへのプッシュ時
on:
  label:
    types:
      - created
  push:
    branches:
      - main
  page_build:

on.<event_name>.types

on.<event_name>.types を使用して、ワークフロー実行をトリガーするアクティビティのタイプを定義します。ほとんどの GitHub イベントは複数種類のアクティビティによってトリガーされます。types キーワードを使用すると、ワークフローの実行を引き起こすアクティビティを絞り込めます。webhook イベントをトリガーするアクティビティタイプが1種類のみの場合、types キーワードは不要です。

on:
  label:
    types: [created, edited]

on.<pull_request|pull_request_target>.<branches|branches-ignore>

pull_request および pull_request_target イベントを使用する場合、特定のブランチをターゲットとするプルリクエストのみでワークフローが実行されるよう設定できます。

ブランチ名パターンを含めたい場合、またはブランチ名パターンの包含と除外の両方を行いたい場合は branches フィルターを使用します。ブランチ名パターンを除外するだけの場合は branches-ignore フィルターを使用します。同一イベントに branches と branches-ignore の両フィルターを使用することはできません。

branches/branches-ignore と paths/paths-ignore の両方を定義した場合、ワークフローは両方のフィルターが満たされた時のみ実行されます。

branches および branches-ignore キーワードは、*、**、+、?、! などのグロブパターン文字を使用できます。これらの文字を含む名前でリテラルマッチを行いたい場合は、各特殊文字を \ でエスケープする必要があります。

例：ブランチを含める

on:
  pull_request:
    # refs/heads に対してマッチするパターンのシーケンス
    branches:
      - main
      - 'mona/octocat'
      - 'releases/**'
ワークフローがブランチフィルター、パスフィルター、またはコミットメッセージによってスキップされた場合、そのワークフローに関連するチェックは「保留中」状態のままになります。それらのチェックが成功することを必須とするプルリクエストはマージがブロックされます。

例：ブランチを除外する

on:
  pull_request:
    # refs/heads に対してマッチするパターンのシーケンス
    branches-ignore:
      - 'mona/octocat'
      - 'releases/**-alpha'

例：ブランチの包含と除外

単一のワークフロー内で同一イベントに branches と branches-ignore の両方を使用することはできません。単一イベントに対してブランチパターンの包含と除外の両方を行いたい場合は、branches フィルターと ! 文字を組み合わせて除外するブランチを指定します。

! 文字を付けてブランチを定義する場合、! を付けないブランチも少なくとも1つ定義する必要があります。ブランチを除外するだけの場合は branches-ignore を使用してください。

パターンの定義順序が重要です。

  • 正のマッチの後にマッチする否定パターン（! プレフィックス）は Git ref を除外します。
  • 否定マッチの後にマッチする正のパターンは Git ref を再度含めます。
on:
  pull_request:
    branches:
      - 'releases/**'
      - '!releases/**-alpha'

on.push.<branches|tags|branches-ignore|tags-ignore>

push イベントを使用する場合、特定のブランチまたはタグでワークフローが実行されるよう設定できます。

ブランチ名パターンを含めたい場合は branches フィルター、除外のみの場合は branches-ignore フィルターを使用します。同一イベントに両方は使用できません。タグも同様に tags と tags-ignore があり、同一イベントに両方は使用できません。

tags/tags-ignore または branches/branches-ignore のいずれか一方のみを定義した場合、未定義の Git ref に影響するイベントではワークフローは実行されません。どちらも定義しない場合、ブランチまたはタグに影響するイベントでワークフローが実行されます。

例：ブランチとタグを含める

on:
  push:
    # refs/heads に対してマッチするパターンのシーケンス
    branches:
      - main
      - 'mona/octocat'
      - 'releases/**'
    # refs/tags に対してマッチするパターンのシーケンス
    tags:
      - v2
      - v1.*

例：ブランチとタグを除外する

on:
  push:
    branches-ignore:
      - 'mona/octocat'
      - 'releases/**-alpha'
    tags-ignore:
      - v2
      - v1.*

例：ブランチとタグの包含と除外

on:
  push:
    branches:
      - 'releases/**'
      - '!releases/**-alpha'

on.<push|pull_request|pull_request_target>.<paths|paths-ignore>

push および pull_request イベントを使用する場合、変更されたファイルパスに基づいてワークフローを実行するよう設定できます。パスフィルターはタグのプッシュには評価されません。

ファイルパスパターンを含めたい場合は paths フィルター、除外のみの場合は paths-ignore フィルターを使用します。同一イベントに両方は使用できません。単一イベントでパスパターンの包含と除外の両方を行いたい場合は、paths フィルターと ! 文字を組み合わせます。

メモ：パスパターンの定義順序が重要です。
  • 正のマッチの後にマッチする否定パターン（! プレフィックス）はパスを除外します。
  • 否定マッチの後にマッチする正のパターンはパスを再度含めます。
branches/branches-ignore と paths/paths-ignore の両方を定義した場合、ワークフローは両方のフィルターが満たされた時のみ実行されます。

paths および paths-ignore キーワードは * と ** のワイルドカード文字を使用したグロブパターンを使用できます。

例：パスを含める

on:
  push:
    paths:
      - '**.js'

例：パスを除外する

on:
  push:
    paths-ignore:
      - 'docs/**'

例：パスの包含と除外

on:
  push:
    paths:
      - 'sub-project/**'
      - '!sub-project/docs/**'

Git diff の比較

メモ：1,000件を超えるコミットをプッシュした場合、またはタイムアウトにより GitHub が diff を生成できない場合は、ワークフローは常に実行されます。

フィルターは、変更されたファイルを評価し、paths-ignore または paths リストと照合することで、ワークフローを実行するかどうかを判断します。変更されたファイルがない場合、ワークフローは実行されません。

GitHub は、プッシュには2ドット diff、プルリクエストには3ドット diff を使用して変更ファイルの一覧を生成します。

  • プルリクエスト：3ドット diff は、トピックブランチの最新バージョンと、トピックブランチがベースブランチと最後に同期されたコミットとの比較です。
  • 既存ブランチへのプッシュ：2ドット diff は、ヘッドとベースの SHA を直接比較します。
  • 新しいブランチへのプッシュ：プッシュされた最も深いコミットの祖先の親に対する2ドット diff。
メモ：diff は300ファイルに制限されています。フィルターが返す最初の300ファイルに一致しない変更ファイルがある場合、ワークフローは実行されません。ワークフローが自動的に実行されるよう、より具体的なフィルターを作成する必要がある場合があります。

on.schedule

on.schedule を使用して、ワークフローのスケジュールを定義できます。

POSIX cron 構文を使用して、特定の時刻にワークフローを実行するようスケジュールします。スケジュールされたワークフローはデフォルトで UTC で実行されます。タイムゾーンを考慮したスケジューリングのために、IANA タイムゾーン文字列を使用してタイムゾーンを指定することもできます。スケジュールされたワークフローはデフォルトブランチの最新コミットで実行されます。スケジュールされたワークフローの最短間隔は5分ごとです。

メモ：夏時間（DST）を採用するタイムゾーンを設定したスケジュールの場合、DST の春の切り替え時に、スキップされた時間のスケジュールされたワークフローは次の有効な時間に進みます。たとえば、午前2:30のスケジュールは3:00 AMに進みます。

cron 構文にはスペースで区切られた5つのフィールドがあり、各フィールドは時間の単位を表します。

┌───────────── 分 (0 - 59)
│ ┌───────────── 時 (0 - 23)
│ │ ┌───────────── 日 (1 - 31)
│ │ │ ┌───────────── 月 (1 - 12 または JAN-DEC)
│ │ │ │ ┌───────────── 曜日 (0 - 6 または SUN-SAT)
│ │ │ │ │
* * * * *
演算子説明例
*任意の値15 * * * * は毎日毎時15分に実行
,値のリスト区切り2,10 4,5 * * * は毎日4時と5時の2分と10分に実行
-値の範囲30 4-6 * * * は4時・5時・6時の30分に実行
/ステップ値20/15 * * * * は20分から59分まで15分ごとに実行（20分、35分、50分）

この例は、America/New_York タイムゾーンで毎週月曜〜金曜の午前5:30にワークフローを実行するようトリガーします。

on:
  schedule:
    - cron: '30 5 * * 1-5'
      timezone: "America/New_York"

単一のワークフローを複数のスケジュールイベントでトリガーできます。github.event.schedule コンテキストを通じてワークフローをトリガーしたスケジュールイベントにアクセスできます。

on:
  schedule:
    - cron: '30 5 * * 1,3'
    - cron: '30 5,17 * * 2,4'

jobs:
  test_schedule:
    runs-on: ubuntu-latest
    steps:
      - name: 月曜と水曜は実行しない
        if: github.event.schedule != '30 5 * * 1,3'
        run: echo "このステップは月曜と水曜にスキップされます"
      - name: 常に実行
        run: echo "このステップは常に実行されます"
