ワークフローの YAML 構文（日本語版） — Part 3

対象範囲：ジョブの実行設定 〜 ステップレベルの構文・マトリックス・再利用可能ワークフロー

本ページは GitHub 公式ドキュメント「ワークフローの構文」の主要キーワードを、分かりやすい日本語でまとめ直した解説です。正確な仕様は必ず公式ドキュメントをご確認ください。

jobs.<job_id>.runs-on（補足）と環境

jobs.<job_id>.environment

environment を使用して、ジョブが参照する環境（デプロイ環境）を指定します。環境には保護ルール（承認者、待機タイマーなど）や環境固有のシークレットを設定でき、デプロイの安全性を高められます。

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://example.com

jobs.<job_id>.concurrency

ジョブ単位で並行実行を制御します。書式はワークフローレベルの concurrency と同じです。

jobs.<job_id>.outputs

outputs を使用して、あるジョブの出力を作成し、依存する後続ジョブ（needs で参照するジョブ）から利用できるようにします。値はステップの出力を参照して設定するのが一般的です。

jobs:
  job1:
    runs-on: ubuntu-latest
    outputs:
      output1: ${{ steps.step1.outputs.test }}
    steps:
      - id: step1
        run: echo "test=hello" >> "$GITHUB_OUTPUT"
  job2:
    needs: job1
    runs-on: ubuntu-latest
    steps:
      - run: echo "${{ needs.job1.outputs.output1 }}"
注意：ジョブの出力はテキストとして扱われ、シークレットを含めるとログにマスクされず露出する恐れがあります。機微な値の受け渡しには使用しないでください。

jobs.<job_id>.steps

ジョブは steps（ステップ）の並びで構成されます。各ステップは、コマンドの実行（run）か、アクションの実行（uses）のいずれかを行います。ステップは上から順番に実行され、同じランナー上で情報を共有できます。

steps[*].id

ステップに一意の ID を付けます。後続のステップから、そのステップの出力やコンテキスト（steps.<id>.outputs.* など）を参照する際に使用します。

steps[*].if

条件が満たされた場合にのみステップを実行します。success()、always()、failure()、cancelled() などのステータス関数と組み合わせて使うことが多いです。

steps:
  - name: 失敗時のみ実行
    if: failure()
    run: echo "前のステップが失敗しました"

steps[*].name

GitHub の UI に表示されるステップ名です。省略可能ですが、ログを読みやすくするために付けることが推奨されます。

steps[*].uses

そのステップで実行するアクションを指定します。再利用可能なコードの単位であるアクションを呼び出すための最も重要なキーワードです。バージョンの固定（タグ、SHA、ブランチ）が強く推奨されます。

steps:
  # 公開アクションをメジャーバージョンで参照
  - uses: actions/checkout@v4
  # コミット SHA で固定（最も安全）
  - uses: actions/checkout@8f4b7f8...
  # 同一リポジトリ内のアクション
  - uses: ./.github/actions/my-action
  # Docker Hub のイメージ
  - uses: docker://alpine:3.18
セキュリティのヒント：サードパーティ製アクションを使う場合は、可変のタグ（例：@v4）ではなく、完全な commit SHA に固定するとサプライチェーン攻撃のリスクを下げられます。

steps[*].run

ランナーのシェルでコマンドを実行します。1 行でも、| を使って複数行でも記述できます。

steps:
  - name: 複数行のコマンドを実行
    run: |
      echo "1行目"
      echo "2行目"

steps[*].working-directory

run コマンドを実行する作業ディレクトリを指定します。

steps:
  - run: npm ci
    working-directory: ./frontend

steps[*].shell

run コマンドで使用するシェルを指定します。bash、pwsh、python、sh、cmd、powershell などを指定できます。

steps[*].with

uses で指定したアクションに渡す入力パラメーターを key: value 形式で指定します。指定できるキーはアクションごとに定義されています。

steps:
  - uses: actions/setup-node@v4
    with:
      node-version: '20'

steps[*].env

その単一ステップでのみ有効な環境変数を設定します。

steps[*].continue-on-error

true にすると、そのステップが失敗してもジョブを失敗扱いにせず処理を続行します。

steps[*].timeout-minutes

ステップを強制終了するまでの最大実行時間（分）を指定します。

jobs.<job_id>.timeout-minutes

ジョブ全体が強制終了されるまでの最大実行時間（分）です。デフォルトは 360 分（6 時間）です。長時間ジョブが課金やランナー占有につながるのを防ぐために設定します。

jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 30

jobs.<job_id>.strategy（マトリックス）

strategy を使用して、ビルドのマトリックスを定義できます。複数の OS や言語バージョンなどの組み合わせで、同じジョブを並行実行するのに便利です。

strategy.matrix

変数とその値の配列を定義すると、すべての組み合わせ分のジョブが自動生成されます。

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}

上記の例では、OS 2 種 × Node 3 種 = 計 6 つのジョブが並行で実行されます。

strategy.fail-fast

デフォルトは true で、マトリックス内のいずれかのジョブが失敗すると、進行中の他のジョブもキャンセルされます。false にするとすべての組み合わせを最後まで実行します。

strategy.max-parallel

マトリックスで同時に実行するジョブの最大数を制限します。

jobs.<job_id>.container

ステップを、ランナー上で直接実行するのではなく、指定した Docker コンテナ内で実行します。依存関係を含んだ独自イメージを使いたい場合に便利です。

jobs:
  build:
    runs-on: ubuntu-latest
    container:
      image: node:20-bookworm
      env:
        NODE_ENV: development

jobs.<job_id>.services

ジョブで利用するサービスコンテナ（データベースやキャッシュなど）を定義します。テスト時に PostgreSQL や Redis を起動する用途でよく使われます。

jobs:
  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: postgres
        ports:
          - 5432:5432

再利用可能ワークフロー（reusable workflows）

jobs.<job_id>.uses

ジョブレベルで uses を使うと、別ファイルに定義した「再利用可能ワークフロー」を呼び出せます。共通の CI 処理を一元管理できます。

jobs:
  call-workflow:
    uses: octo-org/example-repo/.github/workflows/reusable.yml@main
    with:
      config-path: .github/labeler.yml
    secrets:
      token: ${{ secrets.MY_TOKEN }}

呼び出される側：on.workflow_call

再利用可能ワークフロー側では、on: workflow_call を指定し、受け取る inputs や secrets を宣言します。

on:
  workflow_call:
    inputs:
      config-path:
        required: true
        type: string
    secrets:
      token:
        required: true
