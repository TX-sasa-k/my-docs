A. ワークフロー YAML の基本

ワークフローファイルは YAML 構文を使用します。ファイル拡張子は .yml または .yaml でなければなりません。

ワークフローファイルはリポジトリの .github/workflows ディレクトリに格納します。

ワークフローの例（Hello World）

name: Hello World

on:
  push:
    branches: [main]

jobs:
  greet:
    runs-on: ubuntu-latest
    steps:
      - name: Print greeting
        run: echo "Hello, GitHub Actions!"

B. トップレベルキー一覧

キー必須説明
nameワークフローの表示名
run-name各実行（run）の表示名。式が使用可能
on必須ワークフローをトリガーするイベント
permissionsGITHUB_TOKEN に付与する権限
envワークフロー全体で使用できる環境変数
defaultsすべてのジョブに適用されるデフォルト設定
concurrency並行実行の制御
jobs必須ワークフローを構成するジョブ群

C-1. name

ワークフローの名前です。GitHub はリポジトリの「Actions」タブにこの名前を表示します。name を省略した場合、GitHub はリポジトリルートからのワークフローファイルの相対パスを表示します。

name: CI Pipeline

C-2. run-name

ワークフローから生成される実行（run）の名前です。run-name を省略した場合、実行名はワークフローをトリガーしたイベントの情報（コミット メッセージなど）が使用されます。

この値には式（expression）を含めることができ、github および inputs コンテキストを参照できます（ただし env コンテキストは参照できません）。

run-name: Deploy to ${{ inputs.deploy_target }} by @${{ github.actor }}

D. on — トリガー構文

ワークフローを自動的にトリガーするには on を使って、どのイベントでワークフローを実行するかを定義します。単一イベント、複数イベント、スケジュール、手動起動、再利用可能ワークフロー呼び出しなどが設定できます。

単一イベント

on: push

複数のイベント（配列）

on: [push, fork]

複数のイベントを指定した場合、いずれか1つが発生するとワークフローがトリガーされます。同じコミットへの push と pull_request を同時に設定しても、それぞれ独立したワークフロー実行が作成されます。

イベント設定マップ構文

イベントにフィルターやアクティビティタイプを追加するにはマップ構文を使います。

on:
  push:
    branches: [main]
  pull_request:
    types: [opened, synchronize, reopened]
  schedule:
    - cron: '0 0 * * *'

on.<event_name>.types

ワークフロー実行をトリガーするアクティビティのタイプを定義します。多くの GitHub イベントは複数のアクティビティタイプをサポートしています。types を省略した場合は、すべてのアクティビティタイプがワークフローをトリガーします。

on:
  label:
    types: [created, edited]

on:
  issues:
    types: [opened, reopened, labeled]
メモ：複数のアクティビティタイプを設定した場合、ワークフロー実行を作成するためにはそのうち1つ以上が発生する必要があります。複数のアクティビティタイプが同時に発生した場合、複数のワークフロー実行が作成される場合があります。

ブランチ・タグフィルター

pull_request / pull_request_target — ブランチフィルター

pull_request および pull_request_target イベントでは、特定のブランチをターゲットとするプルリクエストのみで実行されるよう設定できます。

  • branches：指定したブランチへの PR のみを含める
  • branches-ignore：指定したブランチへの PR を除外する
注意：branches と branches-ignore を同一イベントに同時指定することはできません。
on:
  pull_request:
    branches:
      - main
      - 'mona/octocat'
      - 'releases/**'

push — ブランチ・タグフィルター

push イベントでは、ブランチとタグの両方でフィルタリングできます。

on:
  push:
    branches:
      - main
      - 'releases/**'
    branches-ignore:
      - 'releases/**-alpha'
    tags:
      - v2
      - v1.*
    tags-ignore:
      - v2-beta*

! を使った包含・除外パターンの順序

ブランチ/タグ名の先頭に ! を付けることで除外パターンを指定できます（branches-ignore とは別の方法）。パターンは宣言した順序で評価されます。

on:
  push:
    branches:
      - 'releases/**'
      - '!releases/**-alpha'   # ← releases/**-alpha は除外
メモ：最後にマッチしたパターンが優先されます。包含パターンの後に除外パターン（!）を書くことで意図した制御が可能です。

パスフィルター

push・pull_request・pull_request_target イベントでは、変更されたファイルパスに基づいてワークフローの実行を制御できます。

  • paths：指定したパスのファイルが変更された場合に実行
  • paths-ignore：指定したパスのファイルだけが変更された場合は実行しない
注意：paths と paths-ignore を同一イベントに同時指定することはできません。
on:
  push:
    paths:
      - '**.js'
      - 'src/**'

on:
  push:
    paths-ignore:
      - 'docs/**'
      - '**.md'

ブランチフィルターとパスフィルターの組み合わせ

on:
  push:
    branches: [main]
    paths: ['src/**', 'tests/**']

上記の場合、両方の条件を満たす場合のみワークフローが実行されます（main ブランチへの push かつ src/ または tests/ 配下のファイルが変更されている）。

Git diff の制限とパスフィルターによるスキップ

メモ（Git diff の制限）：非常に大規模なプルリクエストや、フォークからのプルリクエストで GitHub が変更ファイルの一覧を取得できない場合、パスフィルターは正しく動作せず、ワークフローが意図せず実行・スキップされる場合があります。詳細は公式ドキュメントを参照してください。

on.schedule — スケジュール実行（cron）

POSIX cron 構文を使って、特定の時刻にワークフローを定期実行するようスケジュールできます。スケジュールされたワークフローは UTC タイムゾーンで動作します。

cron フィールドテーブル

フィールド値の範囲説明
分（minute）0–59
時（hour）0–23
日（day of month）1–31
月（month）1–12 または JAN-DEC
曜日（day of week）0–6 または SUN-SAT0 と 7 はどちらも日曜日
┌──────── 分 (0–59)
│ ┌────── 時 (0–23)
│ │ ┌──── 日 (1–31)
│ │ │ ┌── 月 (1–12 または JAN-DEC)
│ │ │ │ ┌ 曜日 (0–6 または SUN-SAT)
│ │ │ │ │
* * * * *
on:
  schedule:
    - cron: '30 5 * * 1,3'      # 毎週月・水 05:30 UTC
    - cron: '30 5,17 * * 2,4'   # 毎週火・木 05:30 と 17:30 UTC
    - cron: '0 0 * * *'         # 毎日 00:00 UTC
メモ：スケジュールの最短間隔は 5 分です。schedule イベントは、GitHub Actions が高負荷になると遅延する場合があります。また、リポジトリがデフォルトブランチで 60 日間操作がない場合は自動的に無効化されます。

on.workflow_dispatch — 手動トリガー

workflow_dispatch を使うと、GitHub UI や API から手動でワークフローを実行できます。inputs を定義することで、実行時に値を渡すこともできます。

サポートされる入力タイプ

タイプ説明
stringテキスト入力
choiceドロップダウン選択。options で選択肢を定義
booleanチェックボックス（true / false）
environmentデプロイ環境を選択するドロップダウン
number数値入力
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'デプロイ先の環境'
        required: true
        type: choice
        options:
          - production
          - staging
          - development
      debug_enabled:
        description: 'デバッグモードを有効化するか'
        required: false
        type: boolean
        default: false
      version:
        description: 'デプロイするバージョン番号'
        required: true
        type: string
        default: 'latest'

inputs コンテキストへのアクセス

ワークフロー内では inputs コンテキスト（または github.event.inputs）で入力値を参照できます。

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Show inputs
        run: |
          echo "環境: ${{ inputs.environment }}"
          echo "デバッグ: ${{ inputs.debug_enabled }}"
          echo "バージョン: ${{ inputs.version }}"

on.workflow_call — 再利用可能ワークフロー

workflow_call を使うと、別のワークフローから呼び出せる再利用可能ワークフロー（reusable workflow）を定義できます。

inputs（入力）

呼び出し元から値を受け取ります。

on:
  workflow_call:
    inputs:
      config-path:
        required: true
        type: string
        description: '設定ファイルのパス'
      node-version:
        required: false
        type: string
        default: '20'

outputs（出力）

呼び出し元のワークフローに値を返します。

on:
  workflow_call:
    outputs:
      workflow_output1:
        description: '呼び出し元に返す値'
        value: ${{ jobs.my-job.outputs.job-output1 }}

secrets（シークレット）

呼び出し元からシークレットを受け取ります。

on:
  workflow_call:
    secrets:
      access-token:
        required: true
      optional-token:
        required: false

再利用可能ワークフローの完全な例（呼び出される側）

# .github/workflows/reusable-deploy.yml
name: Reusable Deploy Workflow

on:
  workflow_call:
    inputs:
      environment:
        required: true
        type: string
    secrets:
      deploy-key:
        required: true
    outputs:
      deployment-url:
        description: 'デプロイ先の URL'
        value: ${{ jobs.deploy.outputs.url }}

jobs:
  deploy:
    runs-on: ubuntu-latest
    outputs:
      url: ${{ steps.deploy-step.outputs.url }}
    steps:
      - id: deploy-step
        run: |
          echo "url=https://example.com" >> "$GITHUB_OUTPUT"
        env:
          DEPLOY_KEY: ${{ secrets.deploy-key }}

再利用可能ワークフローの呼び出し側の例

# .github/workflows/caller.yml
jobs:
  call-deploy:
    uses: octo-org/example-repo/.github/workflows/reusable-deploy.yml@main
    with:
      environment: production
    secrets:
      deploy-key: ${{ secrets.DEPLOY_KEY }}

  post-deploy:
    needs: call-deploy
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deployed to ${{ needs.call-deploy.outputs.deployment-url }}"
シークレットの継承：呼び出し元のすべてのシークレットを再利用可能ワークフローに渡すには secrets: inherit を使用できます。 
jobs:
  call-workflow:
    uses: ./.github/workflows/reusable.yml
    secrets: inherit

E. permissions

GITHUB_TOKEN に付与する権限を変更します。ワークフール全体またはジョブごとに設定できます。指定できる値は read、write、none です。

ショートカット形式

permissions: read-all      # すべてのスコープを読み取り専用に設定
permissions: write-all     # すべてのスコープを書き込みに設定
permissions: {}            # すべての権限を無効化（none）

スコープ別設定

permissions:
  actions: read
  checks: write
  contents: read
  deployments: write
  id-token: write
  issues: write
  packages: read
  pages: write
  pull-requests: write
  repository-projects: read
  security-events: write
  statuses: write

権限スコープ一覧

スコープ説明
actionsGitHub Actions ワークフローの読み書き（キャッシュ、アーティファクト、ワークフロー実行など）
attestationsアーティファクトの Attestation の読み書き
checksチェックランの読み書き
contentsリポジトリコンテンツ（コード、コミット、リリースなど）の読み書き
deploymentsデプロイメントの読み書き
discussionsGitHub Discussions の読み書き
id-tokenOIDC トークンの取得（write が必要）
issuesIssue の読み書き（コメントなど含む）
modelsGitHub Copilot AI モデルへのアクセス
packagesGitHub Packages の読み書き
pagesGitHub Pages の読み書き
pull-requestsプルリクエストの読み書き（レビュー・コメント含む）
repository-projectsGitHub Projects（リポジトリレベル）の読み書き
security-eventsコードスキャン・Dependabot アラートの読み書き
statusesコミットステータスの読み書き
vulnerability-alertsDependabot アラートの読み書き

フォークリポジトリの権限に関する注意

注意（フォークされたリポジトリ）：フォークされたリポジトリからのプルリクエストでは、GITHUB_TOKEN はデフォルトで読み取り専用です。pull-requests: write を明示的に設定しても、フォークからの PR では機密性の高い操作はブロックされる場合があります。

最小権限の例

permissions:
  contents: read     # チェックアウトに必要な最低限の権限

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write  # ジョブ単位で packages への書き込みを追加
    steps:
      - uses: actions/checkout@v4

F-1. env — 環境変数

環境変数のマップです。ワークフロー全体・ジョブ・ステップの3つのレベルで設定できます。より狭いスコープの設定が優先されます（ステップ > ジョブ > ワークフロー）。

# ワークフールレベル
env:
  SERVER: production
  LOG_LEVEL: info

jobs:
  build:
    runs-on: ubuntu-latest
    # ジョブレベル（ワークフロールレベルの同名変数を上書き）
    env:
      LOG_LEVEL: debug
    steps:
      # ステップレベル（さらに上書き）
      - name: Run with custom log level
        env:
          LOG_LEVEL: trace
        run: echo "Log level: $LOG_LEVEL"
メモ：環境変数名は大文字小文字を区別します。また、GITHUB_* および RUNNER_* で始まる名前は GitHub が予約済みです。

F-2. defaults / defaults.run

すべてのジョブに適用されるデフォルト設定のマップです。jobs.<job_id>.defaults で個々のジョブに対して上書きできます。

defaults.run

すべての run ステップに対するデフォルトのシェルと作業ディレクトリを設定します。

defaults:
  run:
    shell: bash
    working-directory: ./scripts

ジョブレベルの defaults.run（ワークフロールレベルを上書き）

defaults:
  run:
    shell: bash
    working-directory: ./

jobs:
  build:
    runs-on: ubuntu-latest
    defaults:
      run:
        shell: pwsh         # このジョブでは PowerShell を使用
        working-directory: ./src

シェルの選択肢

シェルプラットフォーム説明
bashLinux / macOS / WindowsLinux/macOS はデフォルト。Windows は Git Bash 経由
pwshすべてPowerShell Core（クロスプラットフォーム）
pythonすべてPython スクリプトを実行
shLinux / macOSPOSIX sh
cmdWindowsWindows コマンドプロンプト
powershellWindowsWindows PowerShell (Desktop)
カスタムすべてshell: 'perl {0}' のように任意コマンドを指定可能

G. concurrency — 並行実行制御

同じ並行実行グループ（concurrency group）を使うワークフロー実行またはジョブを一度に1つだけに制限します。ワークフロールレベルとジョブレベルの両方で設定可能です。

基本構文

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

グループ名（group）

グループ名は任意の文字列、または式で指定します。グループ名は大文字小文字を区別しません。

  • 使用できるコンテキスト：github、needs、strategy、matrix、inputs、vars

cancel-in-progress

同じグループで別の実行が進行中のとき、その実行をキャンセルするかどうかを指定します。

  • true：進行中の実行をキャンセルして最新の実行を優先
  • false（デフォルト）：進行中の実行が完了するまで新しい実行を待機させる

順序の注意

メモ：並行実行グループ内では、最新の実行（または待機中の実行）が1つだけ保持されます。複数の実行が待機中の場合、後から待機に入った実行が先に待機していた実行を置き換えます。つまり待機するのは常に最後の実行です。

ジョブレベルの concurrency

jobs:
  deploy:
    runs-on: ubuntu-latest
    concurrency:
      group: deploy-${{ github.ref }}
      cancel-in-progress: false   # 本番デプロイは中断させない

デプロイ ワークフローの例

on:
  push:
    branches: [main]

concurrency:
  group: production-deploy
  cancel-in-progress: false

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."

H. jobs

ワークフロー実行は1つ以上のジョブから構成されます。ジョブはデフォルトで並行（parallel）に実行されます。順番に実行するには jobs.<job_id>.needs で依存関係を定義します。各ジョブはランナー環境内で実行されます。

jobs.<job_id>

各ジョブに一意の ID を設定します。

  • 英字または _ で始める
  • 英数字・-・_ のみ使用可能
  • 最大 100 文字
jobs:
  build:          # ← job_id
    runs-on: ubuntu-latest
    steps:
      - run: echo "Building..."

  test_suite:     # ← job_id（アンダースコア使用可）
    runs-on: ubuntu-latest
    steps:
      - run: echo "Testing..."

jobs.<job_id>.name

GitHub の UI に表示されるジョブの名前です。省略した場合は job_id がそのまま表示されます。

jobs:
  build:
    name: Build Application  # UIではこの名前が表示される
    runs-on: ubuntu-latest

jobs.<job_id>.permissions

特定のジョブに対して GITHUB_TOKEN の権限を設定します。トップレベルの permissions を上書きします。書式は同じです。

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      packages: write
      contents: read

jobs.<job_id>.needs

そのジョブを実行する前に正常完了している必要がある他のジョブを指定します。

jobs:
  job1:
    runs-on: ubuntu-latest
  job2:
    needs: job1          # 文字列で1ジョブを指定
    runs-on: ubuntu-latest
  job3:
    needs: [job1, job2]  # 配列で複数ジョブを指定
    runs-on: ubuntu-latest

needs の成功・失敗・スキップの動作

needs のジョブの状態デフォルトの動作
成功（success）依存ジョブが実行される
失敗（failure）依存ジョブはスキップされる
キャンセル（cancelled）依存ジョブはスキップされる
スキップ（skipped）依存ジョブはスキップされる

always() を使って失敗後も実行する

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: exit 1  # 意図的に失敗

  cleanup:
    needs: build
    if: always()   # build が失敗・キャンセルされても実行
    runs-on: ubuntu-latest
    steps:
      - run: echo "Cleanup after failure"

jobs.<job_id>.if

条件が true のときのみジョブを実行します。式では ${{ }} を省略できます（ただし YAML の予約文字には注意）。

YAML の ! に関する注意：YAML では ! はタグ指示子として扱われます。!= などの否定式を使う場合は引用符で囲むか ${{ }} で包む必要があります。 
if: ${{ !startsWith(github.ref, 'refs/tags/') }}
jobs:
  production-deploy:
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest

jobs.<job_id>.runs-on 必須

ジョブを実行するランナーの種類を指定します。文字列・配列・変数・ラベル指定など複数の書式をサポートします。

GitHub ホストランナー

runs-on: ubuntu-latest
runs-on: ubuntu-22.04
runs-on: windows-latest
runs-on: macos-latest
runs-on: macos-14   # Apple Silicon

ラベルを使ったセルフホストランナー

runs-on: self-hosted
runs-on: [self-hosted, linux, x64]
runs-on: [self-hosted, linux, arm64]

グループとラベルの組み合わせ（大規模ランナー）

runs-on:
  group: ubuntu-runners-group
  labels: [ubuntu-20.04-16core]

変数を使う

runs-on: ${{ vars.RUNNER_OS }}

マトリックスを使う

strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]

jobs:
  test:
    runs-on: ${{ matrix.os }}

jobs.<job_id>.environment

ジョブが参照するデプロイ環境を指定します。環境には保護ルール（レビュアーによる承認など）や環境固有のシークレット・変数を設定できます。

# シンプルな形式
environment: production

# name と url を指定する形式
environment:
  name: production
  url: https://example.com

# 式を使ったダイナミックな URL
environment:
  name: ${{ inputs.env_name }}
  url: ${{ steps.deploy.outputs.url }}

jobs.<job_id>.concurrency

ジョブ単位での並行実行制御です。書式はトップレベルの concurrency と同じです。

jobs.<job_id>.outputs

ジョブの出力を定義します。定義した出力は、このジョブに依存する後続ジョブから needs.<job_id>.outputs.<output_name> で参照できます。

警告：ジョブの出力はプレーンテキストとして扱われ、シークレットの値はログにマスクされません。機微な値を出力として渡すことは避けてください。
jobs:
  job1:
    runs-on: ubuntu-latest
    outputs:
      output1: ${{ steps.step1.outputs.test }}
      output2: ${{ steps.step2.outputs.test }}
    steps:
      - id: step1
        run: echo "test=hello" >> "$GITHUB_OUTPUT"
      - id: step2
        run: echo "test=world" >> "$GITHUB_OUTPUT"

  job2:
    runs-on: ubuntu-latest
    needs: job1
    steps:
      - run: echo "${{ needs.job1.outputs.output1 }} ${{ needs.job1.outputs.output2 }}"

jobs.<job_id>.env

そのジョブ内のすべてのステップで使用できる環境変数のマップです。ワークフロールレベルの同名変数を上書きします。

jobs.<job_id>.defaults

そのジョブ内のすべてのステップに適用されるデフォルト設定のマップです。ワークフロールレベルの defaults を上書きします。

jobs.<job_id>.timeout-minutes

ジョブ全体を強制終了するまでの最大実行時間（分）です。デフォルトは 360 分（6 時間）です。

jobs:
  build:
    runs-on: ubuntu-latest
    timeout-minutes: 30  # 30分超えたら強制終了

jobs.<job_id>.continue-on-error

true にすると、そのジョブが失敗してもワークフロー実行全体が失敗にならずに続行します。

jobs:
  build:
    continue-on-error: true
    runs-on: ubuntu-latest

jobs.<job_id>.strategy — マトリックスビルド

ビルドマトリックスを定義します。同一ジョブを複数の変数の組み合わせ（OS・言語バージョンなど）で並行実行できます。

strategy.matrix

strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node: [18, 20, 22]

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node: [18, 20, 22]
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}

上記の例では 3 OS × 3 バージョン = 9 つのジョブが並行実行されます。

strategy.matrix.include — 特定の組み合わせを追加

strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20]
    include:
      - os: windows-latest
        node: 22
        special: true   # この組み合わせにのみ special 変数を追加

strategy.matrix.exclude — 特定の組み合わせを除外

strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node: [18, 20]
    exclude:
      - os: windows-latest
        node: 18   # Windows + Node 18 の組み合わせを除外

strategy.fail-fast

デフォルトは true。マトリックス内のいずれかが失敗すると、進行中の他のジョブをキャンセルします。false にするとすべての組み合わせを最後まで実行します。

strategy:
  fail-fast: false
  matrix:
    os: [ubuntu-latest, windows-latest]

strategy.max-parallel

マトリックスで同時に実行するジョブの最大数を指定します。デフォルトは GitHub がリソースに基づいて自動設定します。

strategy:
  max-parallel: 3
  matrix:
    version: [1, 2, 3, 4, 5, 6]

jobs.<job_id>.container

ジョブのすべてのステップを指定した Docker コンテナ内で実行します。コンテナを指定しない場合、各ステップはランナーマシン上で直接実行されます。

設定項目

キー説明
image使用する Docker イメージ（必須）
credentialsプライベートレジストリの認証情報（username / password）
envコンテナの環境変数
ports公開するポート
volumesマウントするボリューム
optionsdocker create に渡す追加オプション
container:
  image: node:20-bookworm
  credentials:
    username: ${{ secrets.REGISTRY_USER }}
    password: ${{ secrets.REGISTRY_TOKEN }}
  env:
    NODE_ENV: development
  ports:
    - 3000
  volumes:
    - my_docker_volume:/volume_mount
  options: --cpus 1

jobs.<job_id>.services

ジョブで利用するサービスコンテナ（データベース・キャッシュサーバーなど）を定義します。

services:
  postgres:
    image: postgres:16
    env:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: test
    ports:
      - 5432:5432
    options: >-
      --health-cmd pg_isready
      --health-interval 10s
      --health-timeout 5s
      --health-retries 5

  redis:
    image: redis:7
    ports:
      - 6379:6379

コンテナ上で実行するジョブからサービスにアクセスする

jobs:
  test:
    runs-on: ubuntu-latest
    container: node:20
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: postgres
    steps:
      - run: |
          # コンテナ内からは サービス名がホスト名になる
          psql ******postgres/postgres -c "SELECT 1"

jobs.<job_id>.uses — 再利用可能ワークフローの呼び出し

再利用可能ワークフロー（reusable workflow）をジョブとして呼び出します。uses を指定したジョブには steps を含めることができません。

jobs:
  call-workflow:
    uses: octo-org/example-repo/.github/workflows/reusable.yml@main

  call-with-inputs:
    uses: ./.github/workflows/reusable-local.yml
    with:
      config-path: .github/labeler.yml
      environment: production
    secrets:
      token: ${{ secrets.MY_TOKEN }}

  call-with-inherit:
    uses: ./.github/workflows/another.yml
    secrets: inherit  # すべてのシークレットを継承

uses の書式

書式説明
{owner}/{repo}/.github/workflows/{file}.yml@{ref}他のリポジトリのワークフロー
./.github/workflows/{file}.yml同一リポジトリのワークフロー

jobs.<job_id>.steps

ジョブには steps と呼ばれる一連のタスクを含められます（uses で再利用可能ワークフローを呼び出す場合は除く）。各ステップはコマンドの実行・セットアップ・アクションの実行を行います。

I. ステップ構文（jobs.<job_id>.steps[*]）

キー説明
idステップの一意な識別子。コンテキスト参照に使用
if条件式。真のときのみステップを実行
nameGitHub UI に表示されるステップ名
uses実行するアクションを指定
runシェルコマンドを実行
working-directoryrun の作業ディレクトリ
shell使用するシェルを上書き
withアクションへの入力パラメーター
with.argsDocker アクションへの引数
with.entrypointDocker アクションのエントリーポイント上書き
envそのステップ専用の環境変数
continue-on-error失敗してもジョブを続行する
timeout-minutesステップの最大実行時間（分）

jobs.<job_id>.steps[*].id

ステップの一意の識別子です。コンテキストからステップの出力を参照する際に使います（steps.<id>.outputs、steps.<id>.outcome など）。

steps:
  - id: build
    run: make build
  - run: echo "Build result: ${{ steps.build.outcome }}"

jobs.<job_id>.steps[*].if

条件が満たされた場合にのみステップを実行します。式では ${{ }} を省略できます。

steps:
  - name: Only on main
    if: github.ref == 'refs/heads/main'
    run: echo "This is main branch"

  - name: Always run
    if: always()
    run: echo "Always executed"

  - name: On failure
    if: failure()
    run: echo "Previous step failed"

  - name: On cancelled
    if: cancelled()
    run: echo "Job was cancelled"

jobs.<job_id>.steps[*].name

GitHub の UI に表示されるステップ名です。省略した場合は run または uses の内容が表示されます。

jobs.<job_id>.steps[*].uses

ステップとして実行するアクションを選択します。アクションは再利用可能なコードの単位です。

steps:
  # パブリックリポジトリのアクション（バージョン固定）
  - uses: actions/checkout@v4

  # 特定のコミット SHA に固定（最も安全）
  - uses: actions/cache@d632f26e5ce46d4c57c1bb0d5be53f0c2b39c97e

  # 同リポジトリ内のローカルアクション
  - uses: ./.github/actions/my-local-action

  # Docker Hub のイメージ
  - uses: docker://alpine:3.18

  # プライベートレジストリのイメージ
  - uses: docker://ghcr.io/owner/image:tag
セキュリティ：サードパーティ製アクションを使う場合は、可変タグ（@main など）ではなく完全なコミット SHA に固定することを強く推奨します。タグは書き換えられる可能性があります。

jobs.<job_id>.steps[*].run

OS のシェルを使ってコマンドラインプログラムを実行します。

# 単一コマンド
steps:
  - run: npm install

# 複数コマンド（| を使ったブロックスカラー）
  - run: |
      npm run build
      npm test

# 特定のシェルで実行
  - run: |
      Write-Host "PowerShell on Windows"
    shell: pwsh

jobs.<job_id>.steps[*].working-directory

run コマンドの作業ディレクトリを指定します。ジョブの defaults.run.working-directory を上書きします。

- run: npm test
  working-directory: ./frontend

jobs.<job_id>.steps[*].shell

コマンドを実行するシェルを上書き指定します。

- run: |
    import sys
    print(f"Python {sys.version}")
  shell: python

jobs.<job_id>.steps[*].with

uses で指定したアクションへ渡す入力パラメーターのマップです。パラメーター名はアクションの action.yml で定義されます。

steps:
  - uses: actions/setup-node@v4
    with:
      node-version: '20'
      cache: 'npm'

  - uses: actions/upload-artifact@v4
    with:
      name: dist-files
      path: dist/
      retention-days: 5

with.args — Docker アクションへの引数

Docker コンテナアクションの ENTRYPOINT に渡す引数を定義します。

steps:
  - uses: docker://alpine:3.18
    with:
      args: /bin/sh -c "echo hello"

with.entrypoint — Docker アクションのエントリーポイント上書き

steps:
  - uses: ./.github/actions/my-docker-action
    with:
      entrypoint: /bin/sh

jobs.<job_id>.steps[*].env

その単一ステップでのみ有効な環境変数を設定します。ジョブ・ワークフロールレベルの同名変数を上書きします。

steps:
  - name: Deploy
    run: ./deploy.sh
    env:
      DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
      ENV_NAME: production

jobs.<job_id>.steps[*].continue-on-error

true にすると、そのステップが失敗してもジョブを失敗させずに次のステップへ進みます。

steps:
  - name: Optional linting
    continue-on-error: true
    run: npm run lint

jobs.<job_id>.steps[*].timeout-minutes

そのステップを強制終了するまでの最大実行時間（分）です。デフォルトはジョブの timeout-minutes に依存します。

steps:
  - name: Run slow test
    timeout-minutes: 10
    run: npm run integration-test

ステップの実例集

Actions リポジトリのアクションを使う

steps:
  - uses: actions/checkout@v4
    with:
      fetch-depth: 0   # すべての履歴を取得

  - uses: actions/setup-java@v4
    with:
      java-version: '21'
      distribution: 'temurin'

  - uses: actions/cache@v4
    with:
      path: ~/.m2
      key: ${{ runner.os }}-m2-${{ hashFiles('**/pom.xml') }}

ローカルアクション

# .github/actions/greet/action.yml に定義されたローカルアクションを使う
steps:
  - uses: ./.github/actions/greet
    with:
      who-to-greet: 'GitHub'

Docker アクション

steps:
  - name: Run in Docker container
    uses: docker://python:3.12-slim
    with:
      args: python -c "print('Hello from Docker')"

複数ステップの完全な例

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - id: checkout
        name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        id: test
        run: npm test
        env:
          CI: 'true'

      - name: Upload coverage
        if: steps.test.outcome == 'success'
        uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/

      - name: Notify on failure
        if: failure()
        run: echo "CI failed!" >> "$GITHUB_STEP_SUMMARY"
