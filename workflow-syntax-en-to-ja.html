ワークフローの構文 — GitHub Actions（日本語訳）

原文（英語版）: Workflow syntax for GitHub Actions
日本語版: ワークフローの構文

本ドキュメントは、GitHub Actions の英語版「Workflow syntax」リファレンスの構成・見出し・本文を日本語に翻訳したものです。実装時の正確な仕様は必ず公式ドキュメントを参照してください。

この記事について（About YAML syntax for workflows）

ワークフローファイルは YAML 構文を使用し、ファイル拡張子は .yml または .yaml でなければなりません。YAML が初めての方は、「Learn YAML in Y minutes」を読むとよいでしょう。

ワークフローファイルは、リポジトリの .github/workflows ディレクトリに格納する必要があります。

name

ワークフローの名前です。GitHub は、リポジトリの「Actions」タブにこの名前を表示します。name を省略した場合、GitHub はワークフローファイルのリポジトリルートからの相対パスを表示します。

run-name

ワークフローから生成される実行（run）の名前です。GitHub は、リポジトリの「Actions」タブにあるワークフロー実行の一覧に、この実行名を表示します。run-name を省略した場合、実行名はワークフロー実行をトリガーしたイベントに関連する情報に設定されます。

この値には式（expression）を含めることができ、github および inputs コンテキストを参照できます。

run-name: Deploy to ${{ inputs.deploy_target }} by @${{ github.actor }}

on

ワークフローを自動的にトリガーするには、on を使って、どのイベントでワークフローを実行できるかを定義します。

単一または複数のイベントの定義、スケジュールの設定、あるいは特定のファイル・タグ・ブランチの変更に限定した実行などが可能です。

単一イベントを使う

on: push

複数のイベントを使う

on: [push, fork]

複数のイベントを指定した場合、いずれか1つが発生すればワークフローがトリガーされます。

アクティビティタイプを使う

一部のイベントには、ワークフローをいつ実行するかをより細かく制御するためのアクティビティタイプがあります。on.<event_name>.types でトリガーとなるアクティビティタイプを定義します。

on:
  label:
    types:
      - created

フィルターを使う

一部のイベントにはフィルターがあります。たとえば push イベントでは、特定のブランチへのプッシュに限定して実行できます。

on:
  push:
    branches:
      - main
      - 'releases/**'

on.<event_name>.types

ワークフロー実行をトリガーするアクティビティのタイプを定義します。大半の GitHub イベントは複数のアクティビティタイプによってトリガーされます。

on:
  label:
    types: [created, edited]

on.<pull_request|pull_request_target>.<branches|branches-ignore>

pull_request および pull_request_target イベントを使う場合、特定のブランチをターゲットとするプルリクエストのみで実行されるように設定できます。包含には branches、除外には branches-ignore を使います。

on:
  pull_request:
    branches:
      - main
      - 'mona/octocat'
      - 'releases/**'

on.push.<branches|tags|branches-ignore|tags-ignore>

push イベントを使う場合、特定のブランチまたはタグで実行されるように設定できます。

on:
  push:
    branches:
      - main
      - 'releases/**'
    tags:
      - v2
      - v1.*

on.<push|pull_request|pull_request_target>.<paths|paths-ignore>

変更されたファイルパスに基づいて実行するよう設定できます。包含には paths、除外には paths-ignore を使います。

on:
  push:
    paths:
      - '**.js'

on.schedule

POSIX cron 構文を使って、特定の時刻にワークフローを実行するようスケジュールできます。スケジュールされたワークフローは、デフォルトで UTC タイムゾーンで実行されます。

┌───────────── 分 (0 - 59)
│ ┌───────────── 時 (0 - 23)
│ │ ┌───────────── 日 (1 - 31)
│ │ │ ┌───────────── 月 (1 - 12 または JAN-DEC)
│ │ │ │ ┌───────────── 曜日 (0 - 6 または SUN-SAT)
│ │ │ │ │
* * * * *
on:
  schedule:
    - cron: '30 5 * * 1,3'
    - cron: '30 5,17 * * 2,4'

permissions

GITHUB_TOKEN に付与する権限を変更するには permissions を使います。ワークフロー全体、または個々のジョブに対して設定できます。指定できる値は read、write、none です。

permissions:
  actions: read
  contents: write
  issues: write
# すべて読み取り専用
permissions: read-all
# すべての権限を無効化
permissions: {}

env

ワークフロー内のステップで利用できる環境変数のマップです。ジョブ単位（jobs.<job_id>.env）やステップ単位（jobs.<job_id>.steps[*].env）でも設定できます。

env:
  SERVER: production

defaults

ワークフロー内のすべてのジョブに適用されるデフォルト設定のマップです。ジョブ単位（jobs.<job_id>.defaults）でも設定できます。

defaults.run

すべての run ステップに対するデフォルトの shell と working-directory を設定します。

defaults:
  run:
    shell: bash
    working-directory: ./scripts

concurrency

同じ並行実行グループを使うワークフロー実行・ジョブを、一度に1つだけに制限します。cancel-in-progress: true を指定すると、進行中の実行をキャンセルします。

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs

ワークフロー実行は、1つ以上の jobs から構成されます。ジョブはデフォルトで並行に実行されます。順番に実行するには jobs.<job_id>.needs で依存関係を定義します。

jobs.<job_id>

各ジョブに一意の ID を付けます。job_id は文字列で、英字または _ で始め、英数字・-・_ のみを含められます。

jobs.<job_id>.name

GitHub の UI に表示されるジョブ名です。

jobs.<job_id>.permissions

特定のジョブに対して GITHUB_TOKEN の権限を設定します。書式はトップレベルの permissions と同じです。

jobs.<job_id>.needs

そのジョブの実行前に正常完了している必要がある他のジョブを指定します。

jobs:
  job1:
    runs-on: ubuntu-latest
  job2:
    needs: job1
    runs-on: ubuntu-latest
  job3:
    needs: [job1, job2]
    runs-on: ubuntu-latest

jobs.<job_id>.if

条件が満たされた場合にのみジョブを実行します。式では ${{ }} を省略できます。

jobs.<job_id>.runs-on

ジョブを実行するランナーの種類を指定します（必須）。GitHub ホストランナー、大規模ランナー、セルフホストランナーを指定できます。

runs-on: ubuntu-latest
runs-on: [self-hosted, linux, x64]

jobs.<job_id>.environment

ジョブが参照するデプロイ環境を指定します。環境には保護ルールや環境固有のシークレットを設定できます。

environment:
  name: production
  url: https://example.com

jobs.<job_id>.concurrency

ジョブ単位で並行実行を制御します。書式はトップレベルの concurrency と同じです。

jobs.<job_id>.outputs

あるジョブの出力を作成し、依存する後続ジョブから利用できるようにします。

jobs:
  job1:
    runs-on: ubuntu-latest
    outputs:
      output1: ${{ steps.step1.outputs.test }}
    steps:
      - id: step1
        run: echo "test=hello" >> "$GITHUB_OUTPUT"
警告：ジョブの出力はテキストとして扱われ、シークレットを含めるとログにマスクされません。機微な値の受け渡しには使用しないでください。

jobs.<job_id>.env

そのジョブ内のすべてのステップで利用できる環境変数のマップです。

jobs.<job_id>.defaults

そのジョブ内のすべてのステップに適用されるデフォルト設定のマップです。

jobs.<job_id>.steps

ジョブには、steps と呼ばれる一連のタスクを含められます。各ステップはコマンドの実行、セットアップ作業、またはアクションの実行を行います。

jobs.<job_id>.steps[*].id

ステップの一意の識別子です。コンテキストからステップを参照する際に使います。

jobs.<job_id>.steps[*].if

条件が満たされた場合にのみステップを実行します。success()、always()、failure()、cancelled() などのステータス関数と併用できます。

jobs.<job_id>.steps[*].name

GitHub の UI に表示されるステップ名です。

jobs.<job_id>.steps[*].uses

そのステップの一部として実行するアクションを選択します。アクションは再利用可能なコードの単位です。バージョンを固定して使うことを強く推奨します。

steps:
  - uses: actions/checkout@v4
  - uses: ./.github/actions/my-action
  - uses: docker://alpine:3.18
セキュリティ：サードパーティ製アクションは、可変タグではなく完全な commit SHA に固定すると安全性が高まります。

jobs.<job_id>.steps[*].run

オペレーティングシステムのシェルを使ってコマンドラインプログラムを実行します。

steps:
  - run: |
      echo "1行目"
      echo "2行目"

jobs.<job_id>.steps[*].working-directory

run コマンドの作業ディレクトリを指定します。

jobs.<job_id>.steps[*].shell

コマンドを実行するシェルを上書き指定します（bash、pwsh、python、sh、cmd、powershell など）。

jobs.<job_id>.steps[*].with

uses で指定したアクションへ渡す入力パラメーターのマップです。

steps:
  - uses: actions/setup-node@v4
    with:
      node-version: '20'

jobs.<job_id>.steps[*].env

その単一ステップでのみ有効な環境変数を設定します。

jobs.<job_id>.steps[*].continue-on-error

true にすると、そのステップが失敗してもジョブを失敗させずに続行します。

jobs.<job_id>.steps[*].timeout-minutes

そのステップを強制終了するまでの最大実行時間（分）です。

jobs.<job_id>.timeout-minutes

ジョブ全体が強制終了されるまでの最大実行時間（分）です。デフォルトは 360 分です。

jobs.<job_id>.strategy

ビルドのマトリックスを定義します。複数の OS や言語バージョンの組み合わせで同じジョブを並行実行できます。

strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node: [18, 20, 22]

jobs.<job_id>.strategy.fail-fast

デフォルトは true。マトリックス内のいずれかが失敗すると進行中の他のジョブをキャンセルします。

jobs.<job_id>.strategy.max-parallel

マトリックスで同時に実行するジョブの最大数を指定します。

jobs.<job_id>.container

ステップを指定した Docker コンテナ内で実行します。

container:
  image: node:20-bookworm
  env:
    NODE_ENV: development

jobs.<job_id>.services

ジョブで利用するサービスコンテナ（データベースやキャッシュなど）をホストします。

services:
  postgres:
    image: postgres:16
    ports:
      - 5432:5432

jobs.<job_id>.uses

ジョブとして実行する、再利用可能なワークフロー（reusable workflow）ファイルの場所とバージョンを指定します。

jobs:
  call-workflow:
    uses: octo-org/example-repo/.github/workflows/reusable.yml@main
    with:
      config-path: .github/labeler.yml
    secrets:
      token: ${{ secrets.MY_TOKEN }}

jobs.<job_id>.with / secrets

再利用可能ワークフローを呼び出すジョブでは、with で入力を、secrets でシークレットを渡します。呼び出される側は on.workflow_call で受け取る入力・シークレットを宣言します。

on:
  workflow_call:
    inputs:
      config-path:
        required: true
        type: string
    secrets:
      token:
        required: true
