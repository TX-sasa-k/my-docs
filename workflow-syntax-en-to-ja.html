1. YAML ワークフローの基本

GitHub Actions のワークフローは YAML ファイルで定義します。ファイル拡張子は .yml または .yaml で、リポジトリ内の .github/workflows ディレクトリに配置します。このディレクトリにある各 YAML ファイルが 1 つのワークフローとして扱われます。

ワークフローは、イベントによってトリガーされ、1 つ以上のジョブを実行します。各ジョブはランナー上で実行され、ジョブ内には複数のステップを定義できます。ステップではシェルコマンドを実行したり、再利用可能なアクションを呼び出したりできます。

name: CI
run-name: CI for ${{ github.ref_name }} by @${{ github.actor }}

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  contents: read

jobs:
  test:
    name: Test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

2. トップレベルキー

キー説明
nameワークフロー名。Actions タブに表示されます。省略時はワークフローファイルの相対パスが表示されます。
run-name各ワークフロー実行の表示名。github と inputs コンテキストを使った式を含められます。
onワークフローを起動するイベント、スケジュール、手動実行、再利用可能ワークフロー呼び出しを定義します。
permissionsGITHUB_TOKEN の既定権限をワークフロー全体で設定します。ジョブ単位でも上書きできます。
envワークフロー内のすべてのジョブとステップで使える環境変数を定義します。
defaultsワークフロー内のジョブに適用する既定値を定義します。主に defaults.run を使います。
concurrency同じグループに属するワークフロー実行またはジョブの同時実行を制御します。
jobs実行するジョブの集合を定義します。少なくとも 1 つのジョブが必要です。

2.1 name

name はワークフローの名前です。リポジトリの Actions タブでワークフローを識別するために使われます。

name: Deploy production

2.2 run-name

run-name は、ワークフローから生成される個々の実行の名前です。式を含めることができ、実行者、入力値、ブランチ名などを表示名に反映できます。

run-name: Deploy to ${{ inputs.deploy_target }} by @${{ github.actor }}

3. on トリガー構文の詳細

on はワークフローを起動する条件を定義する必須級の中心キーです。単一イベント、複数イベント、イベントごとの設定、スケジュール、手動実行、再利用可能ワークフローとしての呼び出しを定義できます。

3.1 単一イベント

単一のイベントだけで起動する場合は、イベント名を文字列で指定できます。

on: push

3.2 複数イベント

複数イベントのいずれかで起動する場合は配列を使えます。いずれか 1 つのイベントが発生するとワークフローが起動します。

on: [push, fork]

3.3 イベント設定マップ

イベントにフィルターやアクティビティタイプを設定する場合は、on 配下にイベント名をキーとして定義します。設定が必要なイベントと不要なイベントを混在させる場合、すべてをマップ形式で書きます。

on:
  push:
    branches: [main]
  pull_request:
    types: [opened, synchronize]
  workflow_dispatch:

3.4 on.<event_name>.types アクティビティタイプ

一部のイベントには、より細かなアクティビティタイプがあります。たとえば pull_request には opened、synchronize、closed などがあります。types を指定すると、指定したアクティビティタイプの発生時だけワークフローが実行されます。

on:
  label:
    types: [created, edited]

jobs:
  respond:
    runs-on: ubuntu-latest
    steps:
      - run: echo "A label was created or edited"
複数のアクティビティタイプまたは複数イベントを指定すると、1 回のユーザー操作が複数のワークフロー実行を生成する場合があります。イベントとアクティビティタイプの組み合わせを明確にしてください。

3.5 イベントフィルター

イベントによっては、ブランチ、タグ、パスなどでワークフローの実行対象を絞り込めます。フィルターを複数種類組み合わせた場合、基本的には両方の条件を満たす必要があります。

3.6 pull_request / pull_request_target のブランチフィルター

pull_request と pull_request_target では、プルリクエストのベースブランチに対して branches または branches-ignore を使えます。同じイベントで branches と branches-ignore を同時に使うことはできません。包含と除外を同時に表したい場合は、branches に ! で始まる否定パターンを含めます。

on:
  pull_request:
    branches:
      - main
      - 'releases/**'
      - '!releases/**-alpha'

3.7 push のブランチ・タグフィルター

push では branches、branches-ignore、tags、tags-ignore を使えます。ブランチのみ指定した場合、タグへの push では実行されません。タグのみ指定した場合、ブランチへの push では実行されません。どちらも指定しなければ、ブランチまたはタグへの push で実行されます。

on:
  push:
    branches:
      - main
      - 'releases/**'
    tags:
      - v1.*
      - v2.*

3.8 ! を使う包含・除外パターンと順序

branches、tags、paths では ! で始まる否定パターンを含めることができます。パターンは上から順に評価され、後にあるパターンが前の結果を上書きします。否定パターンを使う場合、同じリスト内に少なくとも 1 つの肯定パターンも必要です。

on:
  push:
    branches:
      - 'releases/**'
      - '!releases/**-alpha'
      - 'releases/beta/mona'

この例では releases/** を含め、releases/**-alpha を除外し、最後に releases/beta/mona を再度含めます。

3.9 パスフィルター paths / paths-ignore

push、pull_request、pull_request_target では変更されたファイルパスで実行対象を絞り込めます。paths は一致した変更がある場合に実行し、paths-ignore は一致する変更だけの場合にスキップします。同じイベントで paths と paths-ignore を同時に使うことはできません。

on:
  push:
    paths:
      - 'src/**'
      - '!src/docs/**'
on:
  pull_request:
    paths-ignore:
      - 'docs/**'
      - '**/*.md'

ブランチ・タグフィルターとパスフィルターを組み合わせると、両方の条件を満たす必要があります。たとえば branches と paths を指定した場合、対象ブランチへの変更で、かつ指定パスに一致する変更がある場合にのみ実行されます。

パスフィルターは Git 差分に基づいて評価されます。差分生成には制限があり、変更ファイル数が非常に多い場合や差分を生成できない場合は、ワークフローが常に実行されることがあります。逆に、パスフィルターでスキップされたワークフローは、関連するチェックが pending のまま残る場合があります。

3.10 schedule と POSIX cron

schedule は POSIX cron 構文でワークフローを定期実行します。スケジュールは UTC で評価され、最短間隔は通常 5 分です。スケジュールされたワークフローは、リポジトリの既定ブランチの最新コミットで実行されます。

位置フィールド値例
1分0-5930
2時0-235
3日1-31*
4月1-12 または JAN-DEC*
5曜日0-6 または SUN-SAT1,3
on:
  schedule:
    - cron: '30 5 * * 1,3'
    - cron: '30 5,17 * * 2,4'

jobs:
  scheduled:
    runs-on: ubuntu-latest
    steps:
      - run: echo "This job runs on a schedule"

3.11 workflow_dispatch と入力

workflow_dispatch を使うと、Actions タブ、GitHub CLI、REST API などから手動でワークフローを実行できます。inputs で入力値を定義できます。入力値は inputs コンテキストから参照できます。

代表的な入力タイプには boolean、choice、number、environment、string があります。choice では options を定義します。

on:
  workflow_dispatch:
    inputs:
      deploy_target:
        description: 'デプロイ先環境'
        required: true
        type: choice
        options:
          - staging
          - production
      dry_run:
        description: '実行内容だけ確認する'
        required: false
        default: true
        type: boolean
      retries:
        description: '再試行回数'
        required: false
        default: 1
        type: number

run-name: Deploy ${{ inputs.deploy_target }} by @${{ github.actor }}

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: echo "target=${{ inputs.deploy_target }}, dry_run=${{ inputs.dry_run }}"

3.12 workflow_call 再利用可能ワークフロー

workflow_call を定義したワークフローは、他のワークフローからジョブとして呼び出せる再利用可能ワークフローになります。呼び出される側は入力、出力、シークレットを宣言できます。呼び出す側は jobs.<job_id>.uses を使います。

呼び出される側

name: Reusable build

on:
  workflow_call:
    inputs:
      config-path:
        description: '設定ファイルのパス'
        required: true
        type: string
      run-tests:
        required: false
        default: true
        type: boolean
    secrets:
      token:
        required: true
    outputs:
      artifact-name:
        description: '生成したアーティファクト名'
        value: ${{ jobs.build.outputs.artifact-name }}

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      artifact-name: ${{ steps.meta.outputs.name }}
    steps:
      - id: meta
        run: echo "name=my-artifact" >> "$GITHUB_OUTPUT"

呼び出す側

jobs:
  call-build:
    uses: octo-org/example-repo/.github/workflows/reusable-build.yml@main
    with:
      config-path: .github/build.yml
      run-tests: true
    secrets:
      token: ${{ secrets.BUILD_TOKEN }}

secrets: inherit を使うと、呼び出し元ワークフローで利用可能なシークレットを呼び出し先へ継承できます。ただし組織・リポジトリの境界、セキュリティ要件、最小権限の原則を確認してください。

4. permissions

permissions は GITHUB_TOKEN に付与される権限を制御します。ワークフロー全体のトップレベル、またはジョブ単位の jobs.<job_id>.permissions で指定できます。いずれかの権限を明示すると、明示されていない権限は通常 none になります。

permissions:
  contents: read
  issues: write
  pull-requests: write
# すべての利用可能な権限を読み取りにする
permissions: read-all

# すべての利用可能な権限を書き込みにする
permissions: write-all

# すべての権限を無効化する
permissions: {}
権限用途代表的な値
actionsワークフロー実行や Actions 関連操作。例: 実行のキャンセル。read / write / none
artifact-metadataビルドアーティファクトのメタデータ操作。read / write / none
attestationsアーティファクト attestations の生成や参照。read / write / none
checksチェックラン、チェックスイートの操作。read / write / none
code-qualityコード品質関連データのアップロードなど。read / write / none
contentsリポジトリ内容。例: checkout、コミット一覧、リリース作成。read / write / none
deploymentsデプロイメントの作成や更新。read / write / none
discussionsGitHub Discussions の操作。read / write / none
id-tokenOpenID Connect (OIDC) トークンの取得。クラウド認証でよく使います。write / none
issuesIssue の作成、コメント、ラベル操作など。read / write / none
modelsGitHub Models の推論 API 利用。read / none
packagesGitHub Packages の読み取り・公開。read / write / none
pagesGitHub Pages のビルドやデプロイ。read / write / none
pull-requestsPull request のコメント、ラベル、レビュー関連操作。read / write / none
security-eventsコードスキャンアラートなどセキュリティイベント。read / write / none
statusesコミットステータスの読み書き。read / write / none
vulnerability-alertsDependabot アラートの読み取り。read / none
フォークされたリポジトリからのワークフローでは、通常、書き込み権限を付与できません。管理者が pull request からのワークフローへ書き込みトークンを送る設定を有効にしている場合など、例外があります。

最小権限の例

name: Stale issue management

on:
  schedule:
    - cron: '0 0 * * *'

permissions: {}

jobs:
  stale:
    runs-on: ubuntu-latest
    permissions:
      issues: write
      pull-requests: write
    steps:
      - uses: actions/stale@v10

5. env と defaults

5.1 env

env は環境変数のマップです。トップレベル、ジョブレベル、ステップレベルで定義できます。より狭いスコープの値が、より広いスコープの同名変数を上書きします。トップレベルの env はワークフロー内のすべてのジョブに適用されます。

env:
  SERVER: production
  NODE_ENV: test

jobs:
  example:
    runs-on: ubuntu-latest
    env:
      NODE_ENV: development
    steps:
      - name: Print env
        env:
          NODE_ENV: step-value
        run: echo "$NODE_ENV on $SERVER"

5.2 defaults と defaults.run

defaults は、ワークフローまたはジョブ内のすべての run ステップに適用する既定値を設定します。主なキーは shell と working-directory です。ジョブレベルの既定値はワークフローレベルの既定値を上書きします。

defaults:
  run:
    shell: bash
    working-directory: ./scripts

jobs:
  build:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./build-scripts
    steps:
      - run: ./build.sh

6. concurrency

concurrency は、同じ並行実行グループに属するワークフロー実行またはジョブが同時に複数走らないよう制御します。トップレベルとジョブレベルの両方で使用できます。

group は並行実行グループ名です。cancel-in-progress を true にすると、同じグループで進行中の実行をキャンセルして新しい実行を優先できます。cancel-in-progress には式も使えます。

グループ名は大文字小文字を区別しません。たとえば prod と Prod は同じグループとして扱われます。また、同じグループ内で待機中の実行の順序は保証されません。
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
jobs:
  deploy:
    runs-on: ubuntu-latest
    concurrency:
      group: deploy-${{ inputs.environment }}
      cancel-in-progress: ${{ inputs.environment != 'production' }}
    steps:
      - run: ./deploy.sh

トップレベルでは一般に github、inputs、vars など、ジョブレベルでは github、inputs、vars、needs、strategy、matrix などのコンテキストが利用できます。利用可能なコンテキストはキーごとに異なるため、公式のコンテキスト可用性表を確認してください。

7. jobs と jobs.<job_id>

jobs はワークフロー実行で実行するジョブのマップです。各ジョブには一意の job_id を付けます。job_id は英字または _ で始め、英数字、-、_ を含められます。

7.1 jobs.<job_id>.name

GitHub UI に表示されるジョブ名です。省略時は job_id が表示名として使われます。

jobs:
  test:
    name: Test on Node.js
    runs-on: ubuntu-latest

7.2 jobs.<job_id>.permissions

特定のジョブだけ GITHUB_TOKEN の権限を変更します。トップレベルの permissions よりもジョブ単位の設定がそのジョブに適用されます。

7.3 jobs.<job_id>.needs

ジョブの依存関係を定義します。依存先ジョブが成功してから実行されます。依存先が失敗またはスキップされた場合、通常そのジョブもスキップされます。失敗やスキップにかかわらず実行したい場合は always() を if で使います。

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: make build

  test:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: make test

  summarize:
    if: ${{ always() }}
    needs: [build, test]
    runs-on: ubuntu-latest
    steps:
      - run: echo "summarize results"

7.4 jobs.<job_id>.if

条件が真の場合だけジョブを実行します。if では通常 ${{ }} を省略できますが、式が ! で始まる場合は YAML の予約記法と衝突するため、${{ }} で囲むか引用符・括弧でエスケープしてください。

jobs:
  production-deploy:
    if: github.repository == 'octo-org/octo-repo-prod'
    runs-on: ubuntu-latest
    steps:
      - run: ./deploy.sh
jobs:
  not-for-tags:
    if: ${{ ! startsWith(github.ref, 'refs/tags/') }}
    runs-on: ubuntu-latest
    steps:
      - run: echo "not a tag"

7.5 jobs.<job_id>.runs-on

runs-on はジョブを実行するランナーを指定する必須キーです。GitHub ホストランナー、大規模ランナー、セルフホストランナーを指定できます。値には単一文字列、式を含む文字列、配列、または group / labels を含むマップを使えます。

runs-on: ubuntu-latest
runs-on: [self-hosted, linux, x64, gpu]
on:
  workflow_dispatch:
    inputs:
      chosen-os:
        required: true
        type: choice
        options: [Ubuntu, macOS]

jobs:
  test:
    runs-on: [self-hosted, "${{ inputs.chosen-os }}"]
    steps:
      - run: echo "Hello"
jobs:
  group-runner:
    runs-on:
      group: ubuntu-runners
      labels: ubuntu-20.04-16core
    steps:
      - run: echo "runner group"
分類例説明
GitHub ホスト Linuxubuntu-latest, ubuntu-24.04, ubuntu-22.04GitHub が管理する Linux ランナー。
GitHub ホスト Windowswindows-latest, windows-2025, windows-2022GitHub が管理する Windows ランナー。
GitHub ホスト macOSmacos-latest, macos-14, macos-15GitHub が管理する macOS ランナー。
セルフホストself-hosted と任意ラベル自分で登録したランナー。配列内のすべてのラベルに一致する必要があります。
ランナーグループgroup と labelsOrganization または Enterprise のランナーグループを対象にできます。

7.6 jobs.<job_id>.environment

ジョブが参照するデプロイ環境を指定します。環境には保護ルール、レビュー、待機時間、環境固有のシークレットや変数を設定できます。文字列または name と url のマップで指定できます。

environment: production
environment:
  name: production
  url: https://example.com

7.7 jobs.<job_id>.concurrency

ジョブ単位で並行実行を制御します。デプロイジョブや環境別ジョブの競合防止に有効です。

7.8 jobs.<job_id>.outputs

ジョブの出力を定義し、needs コンテキストを通して後続ジョブから参照できます。値は通常、ステップ出力からマッピングします。

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image: ${{ steps.meta.outputs.image }}
    steps:
      - id: meta
        run: echo "image=ghcr.io/octo/app:sha" >> "$GITHUB_OUTPUT"

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploy ${{ needs.build.outputs.image }}"
ジョブ出力はテキストとして扱われます。シークレットや機微情報をジョブ出力として渡さないでください。

7.9 env / defaults / steps / timeout-minutes

jobs.<job_id>.env はジョブ内のすべてのステップに適用される環境変数です。jobs.<job_id>.defaults はジョブ内の run ステップの既定値です。jobs.<job_id>.steps はジョブ内のタスク列です。jobs.<job_id>.timeout-minutes はジョブ全体の最大実行時間で、既定は 360 分です。

7.10 jobs.<job_id>.strategy と matrix

strategy はマトリックスビルドを定義します。matrix に複数の軸を指定すると、組み合わせごとにジョブが作られます。include で組み合わせや追加プロパティを加え、exclude で特定の組み合わせを除外できます。

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: true
      max-parallel: 3
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20, 22]
        include:
          - os: ubuntu-latest
            node: 22
            experimental: true
        exclude:
          - os: windows-latest
            node: 18
    continue-on-error: ${{ matrix.experimental == true }}
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test
キー説明
matrixジョブを展開する軸と値の集合。
include既存の組み合わせに値を追加、または新しい組み合わせを追加。
excludeマトリックスから特定の組み合わせを除外。
fail-fast既定は true。1 つのマトリックスジョブが失敗したとき、進行中または待機中の関連ジョブをキャンセルするか制御。
max-parallel同時に実行するマトリックスジョブ数の上限。

7.11 jobs.<job_id>.continue-on-error

true の場合、そのジョブが失敗してもワークフロー実行全体を失敗として扱わず続行できます。マトリックスの実験的構成などでよく使います。

7.12 jobs.<job_id>.container

container を指定すると、ジョブのステップを Docker コンテナ内で実行します。Linux ランナーで利用します。文字列でイメージ名を指定するか、image、credentials、env、ports、volumes、options を含むマップを指定できます。

jobs:
  node:
    runs-on: ubuntu-latest
    container:
      image: node:20-bookworm
      credentials:
        username: ${{ github.actor }}
        password: ${{ secrets.github_token }}
      env:
        NODE_ENV: development
      ports:
        - 80
      volumes:
        - my_docker_volume:/volume_mount
        - /data/my_data
        - /source/directory:/destination/directory
      options: --cpus 1
    steps:
      - run: node --version

7.13 jobs.<job_id>.services

services は、ジョブで利用するデータベースやキャッシュなどのサービスコンテナを定義します。サービスごとに image、credentials、env、ports、volumes、options を指定できます。

jobs:
  integration-test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: postgres
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:7
        ports:
          - 6379:6379
    steps:
      - run: npm run test:integration

7.14 再利用可能ワークフローのジョブ: uses / with / secrets

ジョブは通常 runs-on と steps を持ちますが、再利用可能ワークフローを呼び出すジョブでは jobs.<job_id>.uses を使います。この形式では、呼び出し先ワークフローへの入力は with、シークレットは secrets で渡します。

jobs:
  call-workflow:
    uses: octo-org/example-repo/.github/workflows/deploy.yml@v1
    with:
      environment: production
    secrets:
      token: ${{ secrets.DEPLOY_TOKEN }}
jobs:
  call-workflow-inherit:
    uses: octo-org/example-repo/.github/workflows/deploy.yml@main
    secrets: inherit

8. ステップ構文

jobs.<job_id>.steps は、ジョブ内で順番に実行するステップの配列です。ステップは、アクションを実行する uses ステップか、シェルコマンドを実行する run ステップのいずれかを中心に構成します。

8.1 id

ステップの一意識別子です。後続ステップやジョブ出力から steps.<id>.outputs のように参照できます。

8.2 if

ステップを条件付きで実行します。ジョブの if と同様、式やステータス関数を使えます。

steps:
  - name: Run only on main
    if: github.ref == 'refs/heads/main'
    run: echo "main branch"
  - name: Always upload logs
    if: ${{ always() }}
    run: ./upload-logs.sh

8.3 name

GitHub UI に表示されるステップ名です。省略してもステップは実行されますが、ログを読みやすくするため設定が推奨されます。

8.4 uses

アクションを実行します。リポジトリアクション、同一リポジトリ内のローカルアクション、Docker Hub などの Docker コンテナアクションを指定できます。セキュリティ上、サードパーティアクションはタグよりも完全なコミット SHA で固定すると安全性が高まります。

steps:
  - name: Checkout
    uses: actions/checkout@v4

  - name: Local action
    uses: ./.github/actions/my-action

  - name: Docker action
    uses: docker://alpine:3.18

8.5 run

ランナーまたはジョブコンテナ上のシェルでコマンドを実行します。複数行コマンドには YAML のブロック構文を使います。

steps:
  - name: Install and test
    run: |
      npm ci
      npm test

8.6 working-directory

run コマンドを実行する作業ディレクトリをステップ単位で指定します。defaults.run.working-directory よりもステップ単位の指定が優先されます。

steps:
  - run: npm test
    working-directory: ./frontend

8.7 shell

run コマンドを実行するシェルを指定します。プラットフォームごとに既定シェルが異なります。

指定主な対象説明
bashLinux / macOS / WindowsBash を使用。Windows では Git for Windows の bash が使われます。
shLinux / macOSPOSIX shell。
pwshLinux / macOS / WindowsPowerShell Core。
powershellWindowsWindows PowerShell。
cmdWindowsWindows コマンドプロンプト。
python利用可能なランナーPython スクリプトとして実行。
steps:
  - run: Get-ChildItem Env:
    shell: pwsh

8.8 with、with.args、with.entrypoint

with は uses で指定したアクションに渡す入力パラメーターです。入力名はアクションの action.yml で定義されます。Docker コンテナアクションでは args や entrypoint を使える場合があります。

steps:
  - uses: actions/setup-node@v4
    with:
      node-version: '20'
      cache: npm
steps:
  - uses: docker://alpine:3.18
    with:
      entrypoint: /bin/echo
      args: Hello from Docker action

8.9 env

そのステップだけで有効な環境変数を指定します。ステップレベルの env はジョブレベルやワークフローレベルの同名変数を上書きします。

8.10 continue-on-error と timeout-minutes

continue-on-error: true を指定すると、ステップが失敗してもジョブを失敗として扱わず次のステップへ進みます。timeout-minutes はステップを強制終了するまでの最大実行時間です。

steps:
  - name: Experimental check
    continue-on-error: true
    timeout-minutes: 5
    run: ./experimental-check.sh

9.1 主なキー階層

階層主なキー

9.2 複合例

name: Build, test, and deploy
run-name: ${{ github.workflow }} on ${{ github.ref_name }} by @${{ github.actor }}

on:
  push:
    branches:
      - main
      - 'releases/**'
    paths:
      - 'src/**'
      - '.github/workflows/**'
  pull_request:
    types: [opened, synchronize, reopened]
    branches: [main]
  workflow_dispatch:
    inputs:
      environment:
        required: true
        type: choice
        options: [staging, production]

permissions:
  contents: read

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

env:
  NODE_VERSION: '20'

jobs:
  test:
    name: Test (${{ matrix.os }}, Node ${{ matrix.node }})
    runs-on: ${{ matrix.os }}
    permissions:
      contents: read
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest]
        node: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: npm
      - run: npm ci
      - run: npm test

  deploy:
    if: github.ref == 'refs/heads/main'
    needs: test
    runs-on: ubuntu-latest
    environment:
      name: ${{ inputs.environment || 'staging' }}
      url: https://example.com
    concurrency:
      group: deploy-${{ inputs.environment || 'staging' }}
      cancel-in-progress: false
    permissions:
      contents: read
      deployments: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        run: ./deploy.sh
        env:
          TARGET_ENV: ${{ inputs.environment || 'staging' }}
